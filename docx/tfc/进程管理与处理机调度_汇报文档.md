## 二、进程管理基础

### 2.1 进程控制块（PCB）

xv6-riscv 使用 `struct proc` 作为 PCB 的核心数据结构：

```c
struct proc {
  struct spinlock lock;           // 保护进程结构
  enum procstate state;           // 进程状态
  int pid;                        // 进程标识符
  struct proc *parent;            // 父进程指针
  uint64 ctime;                   // 创建时间（ticks）
  
  // 子进程链表（双向）
  struct proc *cnext;             // 子进程链表后继
  struct proc *cprev;             // 子进程链表前驱
  int child_count;                // 直接子进程数量
  
  // 等待队列优化（wait bucket）
  struct waitbucket *wbucket;     // 所属等待桶
  
  // 调度相关字段
  int queue_level;                // MLFQ 队列级别
  int priority;                   // 静态优先级
  int orig_priority;              // 原始优先级（用于 PI 恢复）
  int boost_count;                // 优先级提升计数
  
  // 统计字段
  uint64 wait_time;               // 累计等待时间
  uint64 run_time;                // 累计运行时间
  int sched_count;                // 被调度次数
  
  // ... 其他字段（trapframe, pagetable 等）
};
```

### 2.2 进程状态与转换

xv6 定义了完整的进程状态机：

```
UNUSED ──────┐
    │ allocproc
    ▼
USED ────────┤
    │ userret   │
    ▼           │
RUNNABLE ─────┤◄──────────────────────────────┐
    │ schedule                         │ wakeup
    ▼                                  │
RUNNING ────┤                          │
    │ schedule                        │
    ▼                                  │
ZOMBIE ◄─────┤◄── exit                 │
    │ wait/waitpid                  │
    ▼                                  │
(回收) ─────────────────────────────► SLEEPING
```

### 2.3 进程控制操作

**fork/exec/wait/waitpid** 是进程管理的四大基本操作：

| 操作 | 描述 | 关键实现 |
|------|------|----------|
| fork | 创建子进程，复制地址空间 | `kfork()` 分配 PCB，复制 trapframe 和页表 |
| exec | 执行新程序 | `exec()` 加载 ELF 文件，替换当前进程的地址空间 |
| wait/waitpid | 等待子进程退出 | `kwait()` 遍历子进程链表，收集僵尸子进程 |

### 2.4 wait bucket 优化

原始 `wakeup(chan)` 需要全表扫描 O(NPROC)。我们实现了基于哈希的等待桶机制：

```c
#define NWCHAN 64  // 等待桶数量

struct waitbucket {
  struct spinlock lock;
  struct proc *head;  // 等待队列头
};

// 根据 channel 地址哈希到桶
static struct waitbucket* waitbucket_for(void *chan) {
  return &waittable[((uint64)chan) % NWCHAN];
}
```

**优化效果**：
- 原始：`wakeup()` 需要 O(NPROC) 扫描
- 优化后：O(k)，其中 k 是同桶内进程数

### 2.5 子进程链表优化

使用双向链表管理子进程，避免全表扫描：

```c
// kfork 时加入父进程的子进程链表
np->cprev = 0;
np->cnext = p->cnext;
if (p->cnext) p->cnext->cprev = np;
p->cnext = np;
p->child_count++;

// reparent 时遍历链表而非全表
void reparent(struct proc *p) {
  struct proc *child = p->cnext;
  while (child) {
    child->parent = initproc;
    child = child->cnext;
  }
}
```

## 三、经典调度算法

### 3.1 调度算法全景

| 算法 | 类型 | 特点 | 实现难度 |
|------|------|------|----------|
| RR | 抢占式 | 时间片轮转，公平 | ★☆☆☆☆ |
| FCFS | 非抢占 | 先来先服务，简单 | ★★☆☆☆ |
| SJF | 非抢占 | 短作业优先，最优平均等待 | ★★★☆☆ |
| MLFQ | 抢占式 | 多级反馈，兼顾响应和吞吐 | ★★★★☆ |
| PRIO | 抢占式 | 静态优先级 + aging | ★★★☆☆ |

### 3.2 Round-Robin（时间片轮转）

RR 是 xv6 默认调度算法，每个进程运行固定时间片后被抢占：

```c
void rr_scheduler(void) {
  struct proc *p;
  struct cpu *c = mycpu();
  c->proc = 0;
  
  for (;;) {
    intr_on();
    intr_off();
    int found = 0;
    
    for (p = proc; p < &proc[NPROC]; p++) {
      acquire(&p->lock);
      if (p->state == RUNNABLE) {
        p->state = RUNNING;
        c->proc = p;
        swtch(&c->context, &p->context);
        c->proc = 0;
        found = 1;
      }
      release(&p->lock);
    }
    if (found == 0) asm volatile("wfi");
  }
}
```

**特性**：
- 时间片大小：`TICKSLICE = 1000000 ticks ≈ 10ms`
- 每次时钟中断（which_dev == 2）触发 yield()
- 公平性好，但响应时间依赖于时间片大小

### 3.3 FCFS（先来先服务）

非抢占式调度，选择创建时间最早的进程：

```c
void fcfs_scheduler(void) {
  struct proc *best = 0;
  uint64 min_ctime = UINT64_MAX;
  
  for (p = proc; p < &proc[NPROC]; p++) {
    acquire(&p->lock);
    if (p->state == RUNNABLE && p->ctime < min_ctime) {
      min_ctime = p->ctime;
      best = p;
    }
    release(&p->lock);
  }
  
  if (best) {
    acquire(&best->lock);
    best->state = RUNNING;
    swtch(&c->context, &best->context);
    release(&best->lock);
  }
}
```

**测试结果**（fcfstest）：
```
Child 0 (PID=4) finished at tick 335 (took 3 ticks)
Child 1 (PID=5) finished at tick 335 (took 3 ticks)
Child 2 (PID=6) finished at tick 338 (took 5 ticks)
Child 3 (PID=7) finished at tick 339 (took 5 ticks)
```

**分析**：完成顺序与 PID 顺序一致，验证 FCFS 的先来先服务特性。

### 3.4 MLFQ（多级反馈队列）

MLFQ 结合了短作业优先和公平性的优点：

```c
// 5级队列设计
#define MLFQ_LEVELS 3
#define MLFQ_Q0_TIME 500000   // Q0: 5ms
#define MLFQ_Q1_TIME 1000000  // Q1: 10ms
#define MLFQ_Q2_TIME 2000000  // Q2: 20ms

void mlfq_scheduler(void) {
  struct proc *best = 0;
  int best_level = MLFQ_LEVELS;
  
  // 选择最高优先级队列中的最早进程
  for (p = proc; p < &proc[NPROC]; p++) {
    acquire(&p->lock);
    if (p->state == RUNNABLE && 
        p->queue_level < best_level) {
      best_level = p->queue_level;
      best = p;
    }
    release(&p->lock);
  }
  
  if (best) {
    acquire(&best->lock);
    best->state = RUNNING;
    best->last_sched = ticks;
    swtch(&c->context, &best->context);
    release(&best->lock);
  }
}
```

**降级机制**（trap.c 时钟中断）：
```c
if (which_dev == 2) {
  p->timeslice_used++;
  int ts = get_timeslice(p->queue_level);
  if (p->timeslice_used >= ts) {
    if (p->queue_level < MLFQ_LEVELS - 1)
      p->queue_level++;  // 降级
    p->timeslice_used = 0;
    yield();
  }
}
```

**优先级提升**（防止饥饿）：
```c
void mlfq_boost_priority(void) {
  for (p = proc; p < &proc[NPROC]; p++) {
    acquire(&p->lock);
    if (p->state == RUNNABLE || p->state == RUNNING) {
      if (p->queue_level > 0)
        p->queue_level--;  // 提升一级
    }
    release(&p->lock);
  }
}
```

**测试结果**（mlfqtest）：
```
SHORT jobs:  elapsed=1~3 ticks  ✓ 短作业最快完成
LONG jobs:   elapsed=5~6 ticks  ✓ 中等时间
MIXED jobs:  elapsed=12 ticks  ✓ 混合作业最慢
Total time:  15 ticks
```

**MLFQ vs FCFS vs RR 对比**：

| 维度 | RR | FCFS | MLFQ |
|------|-----|------|------|
| 公平性 | ★★★★★ | ★★★☆☆ | ★★★★☆ |
| 短作业响应 | ★★★☆☆ | ★★☆☆☆ | ★★★★★ |
| 实现复杂度 | ★☆☆☆☆ | ★★☆☆☆ | ★★★★☆ |
| CPU 利用率 | ★★★★☆ | ★★★★★ | ★★★★★ |

### 3.5 SJF（短作业优先）

非抢占式调度，选择估计执行时间最短的进程：

```c
void sjf_scheduler(void) {
  struct proc *best = 0;
  uint64 min_burst = UINT64_MAX;
  
  for (p = proc; p < &proc[NPROC]; p++) {
    acquire(&p->lock);
    if (p->state == RUNNABLE && 
        p->est_burst < min_burst) {
      min_burst = p->est_burst;
      best = p;
    }
    release(&p->lock);
  }
  // ... 选择 best 运行
}
```

**测试结果**（sjfbusy）：
```
[child 3] est=1 work=1M FINISHED at tick 48  ← 最短
[child 2] est=2 work=2M FINISHED at tick 49
[child 1] est=4 work=4M FINISHED at tick 50
[child 0] est=8 work=8M FINISHED at tick 51  ← 最长
```

**关键发现**：burst 注册必须"提前可见"，否则 SJF 会退化为 FCFS。使用 sem barrier 确保所有进程先注册再切换算法。

## 四、死锁专题：从复现到恢复

### 4.1 死锁的四个必要条件

死锁产生必须同时满足以下四个条件：

| 条件 | 描述 | 哲学家就餐场景 |
|------|------|---------------|
| 互斥 | 资源一次只能被一个进程持有 | 叉子不能同时被两个哲学家使用 |
| 占有并等待 | 进程持有资源同时等待其他资源 | 哲学家拿到左叉后等待右叉 |
| 不可剥夺 | 资源不能被强制夺走 | 叉子不能从哲学家手中抢走 |
| 循环等待 | 形成进程等待环路 | P0→P1→P2→P3→P4→P0 |

### 4.2 B1: 死锁复现 — 哲学家就餐

**测试程序**：`user/dining.c`

```c
#define NPHIL 5
for (int i = 0; i < NPHIL; i++) {
    sem_wait(forks[left(i)]);      // 全部成功拿到左叉
    pause(5);                      // 对齐进度
    sem_wait(forks[right(i)]);      // 全部阻塞在右叉
}
```

**实测 trace**：
```
[phil 0] take LEFT  fork=0
[phil 1] take LEFT  fork=1
[phil 2] take LEFT  fork=2
[phil 3] take LEFT  fork=3
[phil 4] take LEFT  fork=4
[phil 0] try  RIGHT fork=1 -> BLOCKED
[phil 1] try  RIGHT fork=2 -> BLOCKED
...
[DEADLOCK] 5 processes sleeping forever
```

**结论**：5 个进程 100% 稳定死锁，qemu timeout 强制终止。

### 4.3 B2: 死锁预防 — 破坏死锁条件

#### 方案 A: 破坏"占有并等待"

**原理**：限制同时竞争资源的进程数。

```c
// 最多 4 个哲学家同时入座
sem_wait(room);           // 先抢"入场票"
sem_wait(fork[left]);
sem_wait(fork[right]);
eat();
sem_post(fork[right]);
sem_post(fork[left]);
sem_post(room);
```

**数学保证**：5 把叉子 + 最多 4 个竞争者 → 必有 1 个能拿到 2 把叉子。

**测试结果**：`dining_safe1` 连续 1000 轮无死锁。

#### 方案 B: 破坏"循环等待"

**原理**：资源全序编号，强制按顺序获取。

```c
int first = min(i, (i+1)%N);
int second = max(i, (i+1)%N);
sem_wait(fork[first]);
sem_wait(fork[second]);
```

**数学保证**：全序关系下不可能形成有向环。

**测试结果**：`dining_safe2` 连续 1000 轮无死锁，并发度 5/5（优于方案 A 的 4/5）。

### 4.4 B3: 死锁避免 — 银行家算法

**理论背景**（Silberschatz Ch7）：每次资源分配前，模拟分配并运行安全性算法，确保存在安全序列才真正分配。

**核心数据结构**：
```c
struct banker_state {
  int available[NRES];           // 可用资源向量
  int max[NPROC_B][NRES];       // 每个进程最大需求
  int allocation[NPROC_B][NRES]; // 当前已分配
  int need[NPROC_B][NRES];      // 仍需资源 = max - allocation
};
```

**安全性算法**：
```c
int is_safe(int *out_seq) {
  int work[NRES] = {available};
  int finish[NPROC_B] = {0};
  
  while (found) {
    found = 0;
    for each i where finish[i]==0 && need[i]<=work {
      work += allocation[i];  // 回收 i 的资源
      finish[i] = 1;
      found = 1;
    }
  }
  return (all finish[i]==1) ? 0 : -1;
}
```

**教科书 T0 状态验证**：

| 进程 | Allocation | Max | Need | Available |
|------|-----------|-----|------|-----------|
| P0 | (0,1,0) | (7,5,3) | (7,4,3) | (3,3,2) |
| P1 | (2,0,0) | (3,2,2) | (1,2,2) | ← 从 P1 开始安全 |
| P2 | (3,0,2) | (9,0,2) | (6,0,0) | |
| P3 | (2,1,1) | (2,2,2) | (0,1,1) | |
| P4 | (0,0,2) | (4,3,3) | (4,3,1) | |

安全序列：`<P1, P3, P4, P2, P0>` — 与 Silberschatz 完全一致。

**实测 trace**：
```
[banker] init OK (nres=3, available=(3,3,2))
[banker] initial safe sequence: <P1, P3, P4, P0, P2>
Test: P0 request (0,2,0) -> GRANTED (safe)
Test: P0 request (3,3,2) -> REFUSED (unsafe)
=== Banker Test PASSED ===
```

### 4.5 B4: 死锁检测 + 自动恢复

**原理**：运行时周期性检测等待图中的环，发现死锁后剥夺最年轻进程。

**等待图构建**：
```c
// 若进程 i 等待 sem[j]，而 sem[j] 被进程 h 持有，则建边 i -> h
for (int i=0; i<n; i++)
  for (int j=0; j<n; j++)
    if (i!=j && procs[i].chan==&semtable[j] && 
        semtable[j].holder_pid==procs[j].pid)
      add_edge(i, j);
```

**DFS 环检测**：标准图论算法，检测 back-edge。

**实测 trace**：
```
[DEADLOCK] cycle detected, length=3: pid=4 -> pid=5 -> pid=4
[DEADLOCK] aborting victim pid=5 (youngest in cycle)
[DEADLOCK] releasing sem=0..4 (was held by pid=5)
[sem_wait] pid=4 sem=1 -> woke up
[phil 0] EATING  (round 0)
[phil 2] EATING  (round 0)
=== 剩余 4 个哲学家继续全并发用餐 ===
```

### 4.6 死锁处理策略对比

| 策略 | 预防 | 避免 | 检测+恢复 |
|------|------|------|----------|
| 破坏条件 | 破坏 1 个即可 | — | — |

## 五、高级同步原语

### 5.1 信号量回顾

信号量是 Dijkstra 提出的经典同步原语：

| 操作 | 语义 | 实现 |
|------|------|------|
| sem_open(val) | 创建信号量，初始值 = val | 分配 semaphore 结构 |
| sem_wait(P) | 原子递减，若 < 0 则阻塞 | value--; if < 0 sleep() |
| sem_post(V) | 原子递增，唤醒等待者 | value++; wakeup() |

**生产者-消费者示例**（semtest3）：
```c
// 3 个信号量协同
empty = sem_open(BUF);  // 初始 5 个空槽
full = sem_open(0);     // 初始 0 个满槽
mutex = sem_open(1);    // 互斥

producer() {
  sem_wait(empty);
  sem_wait(mutex);
  buf[in] = item; in = (in+1)%BUF;
  sem_post(mutex);
  sem_post(full);
}

consumer() {
  sem_wait(full);
  sem_wait(mutex);
  item = buf[out]; out = (out+1)%BUF;
  sem_post(mutex);
  sem_post(empty);
}
```

### 5.2 C1: 管程（Monitor）

**理论背景**：管程是一种高级同步抽象，将共享资源和对资源的操作封装在一起。

**Mesa 语义**（xv6 采用）：
- signal 后 waiter 进入就绪队列，不直接运行
- 醒来后需要重新获取管程锁

**核心实现**：
```c
struct monitor {
  char name[16];
  struct spinlock lock;    // 管程互斥锁
  int owner;               // 当前持有者 PID
  struct cv {
    int signaled;          // Mesa 风格标记
    struct spinlock lock;
    struct proc *waiters;
  } cond[NCOND];
};

monitor_wait(mid, cvid):
  sem_post(mutex);    // 1. 释放管程锁
  sem_wait(cv);       // 2. 睡在条件变量上
  sem_wait(mutex);    // 3. 醒来后重新获得锁

monitor_signal(mid, cvid):
  sem_post(cv);       // 唤醒一个 waiter
```

**测试结果**（monitortest）：
```
[prod 1] put 102 at 2  (count=3)   <- 写满
[cons 1] got 100 from 1  (count=2)  <- 消费后 signal
[cons 1] got 101 from 2  (count=1)
=== Monitor test PASSED ===
```

### 5.3 C2: 管程重写生产者-消费者

**挑战**：xv6 fork() 不共享内存，静态变量在 fork 后各自独立。

**解决方案**：使用共享内存传递状态。

```c
// 共享数据结构
ctrl = shmat(shmget(0x1234, sizeof(int)*5, 0));
buf  = shmat(shmget(0x1235, BUF*sizeof(int), 0));

// 用管程替代 3 个信号量
mon_lock(m);
while (count == BUF) mon_wait(m, 0);  // not_full
buf[in] = item; in = (in+1)%BUF; count++;
mon_signal(m, 1);                      // not_empty
mon_unlock(m);
```

**信号量 vs 管程对比**：

| 维度 | 信号量 | 管程 |
|------|--------|------|
| 同步原语数 | 3（empty/full/mutex） | 1 monitor + 2 CV |
| 代码复杂度 | 较高（需手动维护计数） | 较低（计数由 CV 隐式管理） |
| 错误风险 | 需正确配对 wait/post | 结构更清晰 |
| 适用场景 | 底层原语 | 高层抽象 |

### 5.4 D1: 优先级继承（Priority Inheritance）

**问题背景**：1997 年火星探测器真实故障。

**问题场景**：
```
L (low)  持锁 ──→ M (medium) busy-loop ──→ H (high) 等待锁
```

无 PI：M 抢占 L，L 无法释放锁，H 永远等待。
有 PI：H 阻塞时 L 临时提升到 H 优先级，M 无法抢占，L 完成，H 继续。

**内核实现**（kernel/sem.c）：
```c
// sem_wait 中优先级继承
if (sem->value < 1 && sem->holder_pid >= 0) {
  holder->priority = min(holder->priority, my_priority);
  holder->boost_count++;
}

// sem_post 中优先级恢复（关键：移到 if/else 外！）
sem->value++;
if (sem->value <= 0)
  wakeup(sem);
if (p->boost_count > 0) {
  p->boost_count--;
  if (p->boost_count == 0)
    p->priority = p->orig_priority;
}
```

**实测 trace**（pathfinder）：
```
[L  pid=4] acquired sem, doing critical work
[M  pid=5] busy-looping
[L  pid=4] releasing sem, prio=0    <- L 被提升到 H 的优先级
[L  pid=4] done
[H  pid=6] got sem after 7028888 us
[H  pid=6] done
=== Phase D1 PASSED ===
```

### 6.3 F1: Rate-Monotonic（RM）

**理论**（Liu & Layland 1973）：静态优先级 = 周期倒数。

**可调度性条件**：
```
∑(Ci/Ti) ≤ n(2^(1/n) - 1)
```

| n | 理论上限 |
|---|---------|
| 1 | 1.000 |
| 2 | 0.828 |
| 3 | 0.779 |
| ∞ | 0.693 |

**测试配置**（利用率 = 0.60 < 0.779）：

| 任务 | 周期 Ti | 执行时间 Ci | 利用率 |
|------|---------|------------|--------|
| T1 | 10 | 2 | 0.20 |
| T2 | 20 | 4 | 0.20 |
| T3 | 40 | 8 | 0.20 |

### 6.4 F2: Earliest Deadline First（EDF）

**理论**：动态优先级 = 截止时间最近。

**可调度条件**：∑(Ci/Ti) ≤ 1（理论上可达 100% 利用率）。

```c
struct proc *edf_scheduler(void) {
  struct proc *best = 0;
  for (p = proc; p < &proc[NPROC]; p++) {
    if (p->state == RUNNABLE && IS_RT_TASK(p) &&
        (!best || p->rt_deadline < best->rt_deadline))
      best = p;
  }
  return best;
}
```

### 6.5 RM vs EDF 对比

| 维度 | RM | EDF |
|------|----|-----|
| 优先级类型 | 静态 | 动态 |
| 可调度上限 | ≈ 0.693 | 1.0（100%） |
| 实现复杂度 | 低 | 中 |
| 适用场景 | 硬实时周期任务 | 动态截止时间任务 |

### 6.6 E1: Per-CPU 亲和性机制

**目标**：进程固定到特定 CPU，支撑 per-CPU 调度队列和负载均衡。

```c
// struct proc 新增字段
int cpu_affinity;  // -1=无偏好, 0..NCPU-1=固定 CPU

// 两遍选择算法
// 第一遍：选 cpu_affinity 匹配当前 CPU 或无偏好的最优进程
// 第二遍：若无可用亲和进程，选任意可运行进程（防止 CPU 空闲）
```

**限制**：QEMU `-bios none -kernel` 只拉起 hart 0，机制已就位但无法演示跨 CPU 迁移。

## 九、关键 Bug 与调试经验

### 9.1 关键 Bug 总结

| # | Bug 描述 | 影响 | 修复 |
|---|----------|------|------|
| 1 | `reparent()` 中 `cnext` 被误用为子进程链表（实际是兄弟链表） | 多子进程 exit 时 wait 永久阻塞 | O(N) 扫描 proc[] 数组 |
| 2 | PI 中 `sem_post` 只在无 waiter 分支恢复优先级 | PI boost 永不释放，优先级反转永久化 | 移到 if/else 外 |
| 3 | `banker.c` 中 `recompute_need` 在 `nproc` 更新前调用 | Need 全零，is_safe 误判 | 调整调用顺序 |
| 4 | xv6 `fork()` 不共享内存，`pc_monitor` 用静态变量死锁 | monitor 无效，4 个进程全睡在 CV 上 | 改用 shmget/shmat 共享状态 |
| 5 | `freeproc` 中 shm PTE 未清理，`freewalk` panic | 子进程 exit 时 kernel panic | 在 proc_freepagetable 前显式 unmap shm leaf PTE |

### 9.2 调试经验

#### 经验 1: xv6 scheduler 单次缓存陷阱

**问题**：`scheduler()` for(;;) 循环里缓存算法到局部变量，切换不生效。

**修复**：直接读全局变量：
```c
// 修复前
for(;;) {
  int algo = current_scheduler;  // 缓存
  if (algo == SJF) { ... }
}

// 修复后
for(;;) {
  if (current_scheduler == SJF) { ... }  // 直接读全局
}
```

#### 经验 2: 调度算法切换 = 立即 yield

只改 flag 不 yield，当前进程不会被新算法选中：
```c
current_scheduler = algo;
yield();  // 关键！让调度器用新算法重选下一个进程
```

#### 经验 3: SJF 的 burst 必须"提前可见"

`sched_setburst` 在 child 第一次被调度时才执行，但 SJF 选择先于 child 第一次被调度。

**解法**：使用 sem barrier 让所有 child 先注册 burst，再切到 SJF：
```c
// MLFQ 模式下 fork + 注册 burst + sem_wait SLEEPING
for (int i = 0; i < 4; i++) {
  pid = fork();
  if (pid == 0) {
    sched_setburst(getpid(), est[i]);  // 先设 burst
    sem_wait(sem_barrier_id);          // 阻塞等释放
  }
}
// 所有 child 注册 burst 后再切到 SJF
sched_algorithm(3);
for (int i = 0; i < 4; i++) sem_post(sem_barrier_id);
```

#### 经验 4: PI 优先级恢复必须在 if/else 外

```c
// 修复前（只在无 waiter 分支恢复）
if (sem->value <= 0) {
  wakeup(sem);
} else {
  // 这里恢复，但通常 post 是因为有 waiter
  p->priority = p->orig_priority;
}

// 修复后（无论有无 waiter 都恢复）
sem->value++;
if (sem->value <= 0)
  wakeup(sem);
if (p->boost_count > 0) {
  p->boost_count--;
  if (p->boost_count == 0)
    p->priority = p->orig_priority;
}
```
