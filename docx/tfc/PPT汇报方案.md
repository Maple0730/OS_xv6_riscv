### 第 5 页 — 进程控制操作

**标题**：进程控制：fork / exec / wait / waitpid

**内容**：

| 操作 | 功能 | 关键实现 |
|------|------|----------|
| fork | 创建子进程 | `kfork()` 分配 PCB，复制页表 |
| exec | 执行新程序 | `exec()` 加载 ELF，替换地址空间 |
| wait | 等待任意子进程 | 遍历子进程链表，收集僵尸 |
| waitpid | 等待指定 PID | 按 PID 精确匹配 |

**wait bucket 优化**：
```
原始 wakeup(chan): O(NPROC) 全表扫描
优化后 wakeup(chan): O(k) 哈希桶定位
```

**代码片段**：
```c
// wait bucket 哈希机制
static struct waitbucket* waitbucket_for(void *chan) {
  return &waittable[((uint64)chan) % NWCHAN];
}
```

- [图片7] wait bucket 优化前后对比图

**讲解词**：
> 进程控制四大操作：fork 创建子进程、exec 执行新程序、wait 和 waitpid 等待子进程退出。我们还实现了 wait bucket 优化——原始的 wakeup 需要遍历所有进程，我们用哈希桶机制把复杂度从 O(NPROC) 降到 O(k)。这是第一个优化点。

### 第 8 页 — MLFQ 核心实现

**标题**：MLFQ：兼顾响应与吞吐

**内容**：

**队列设计**：
```
  Q0 (最高优先级) ── 时间片: 5ms  ── 交互式进程
       │
       ▼
  Q1 (中等优先级) ── 时间片: 10ms ── 普通进程
       │
       ▼
  Q2 (最低优先级) ── 时间片: 20ms ── CPU 密集型
```

**降级与提升**：
```
降级: 时间片用完 → queue_level++
提升: 每 100 ticks → 所有进程 queue_level = 0
```

**关键代码**：
```c
// 时钟中断处理
if (p->timeslice_used >= ts) {
  if (p->queue_level < MLFQ_LEVELS - 1)
    p->queue_level++;  // 降级
  yield();
}
```

**占位符说明**：
- [图片12] MLFQ 队列状态变化示意图
- [图片13] 时间片用完降级代码截图

**讲解词**：
> MLFQ 是最复杂的调度算法。它维护 3 个优先级队列：高队列时间片 5ms，适合交互式进程；中队列 10ms；低队列 20ms，适合 CPU 密集型。关键机制有两个：一是降级——用完时间片后降到更低队列；二是提升——每 100 个时钟 tick，所有进程回到最高队列，防止低优先级进程饥饿。

## 第5部分：死锁专题

### 第 10 页 — 死锁四条件复现

**标题**：死锁复现：哲学家就餐问题

**内容**：

**死锁四条件**：
| 条件 | 在哲学家就餐中的体现 |
|------|---------------------|
| 互斥 | 叉子一次只能被一个哲学家持有 |
| 占有并等待 | 拿到左叉后等待右叉 |
| 不可剥夺 | 叉子不能强制夺走 |
| 循环等待 | P0→P1→P2→P3→P4→P0 |

**复现实验**：
```c
// 5个哲学家全部先拿左叉
for (int i = 0; i < 5; i++) {
  sem_wait(forks[left(i)]);   // 全部成功
  pause(5);                   // 对齐进度
  sem_wait(forks[right(i)]);   // 全部阻塞
}
```

**实测日志**：
```
[phil 0] take LEFT  fork=0
[phil 1] take LEFT  fork=1
...
[phil 0] try  RIGHT fork=1 -> BLOCKED
[phil 1] try  RIGHT fork=2 -> BLOCKED
...
[DEADLOCK] 5 processes sleeping forever
```

**占位符说明**：
- [图片16] dining 程序运行截图（5 个 SLEEPING）
- [图片17] 哲学家就餐死锁示意图

**讲解词**：
> 死锁是进程同步中最经典的问题。我们用哲学家就餐问题来复现死锁。右表列出了死锁的四个必要条件——互斥、占有并等待、不可剥夺、循环等待。测试程序让 5 个哲学家同时拿左叉后等待右叉，实测日志显示 5 个进程全部进入 SLEEPING，完美复现了死锁。

### 第 11 页 — 死锁预防：两种方案

**标题**：死锁预防：破坏循环等待 vs 破坏占有并等待

**内容**：

**方案 A：破坏"占有并等待"**
```
原理: 限制同时竞争资源的进程数

实现: room 信号量（init=4）
      最多4个哲学家同时入座

数学保证: 5把叉子 + 最多4人 → 必有1人能拿到2把
```

**方案 B：破坏"循环等待"**
```
原理: 资源全序编号，强制按顺序获取

实现: 奇数号先左后右，偶数号先右后左

数学保证: 全序关系下不可能形成有向环
```

**对比**：

| 方案 | 破坏条件 | 并发度 | 代码复杂度 |
|------|----------|--------|------------|
| A (room) | 占有并等待 | 4/5 | 低 |
| B (全序) | 循环等待 | **5/5** | 中 |

**实测**：`dining_safe1` 和 `dining_safe2` 连续 1000 轮无死锁。

**占位符说明**：
- [图片18] dining_safe1 和 dining_safe2 运行截图
- [图片19] 两种方案对比图

**讲解词**：
> 死锁预防的核心是破坏四个必要条件之一。方案 A 破坏"占有并等待"——引入 room 信号量限制同时入座的哲学家数为 4，这样 5 把叉子中必有 1 把空闲。方案 B 破坏"循环等待"——通过资源全序编号，奇偶号哲学家取叉顺序不同，彻底消除环路。全序方案达到 5/5 并发度，优于方案 A 的 4/5。

### 第 12 页 — 银行家算法

**标题**：死锁避免：银行家算法

**内容**：

**算法思想**：
```
每次资源分配前:
1. 模拟分配请求
2. 运行安全性算法
3. 存在安全序列 → 允许分配
   不存在       → 拒绝请求
```

**教科书 T0 状态**：

| 进程 | Allocation | Max | Need | Available |
|------|-----------|-----|------|-----------|
| P0 | (0,1,0) | (7,5,3) | (7,4,3) | (3,3,2) |
| P1 | (2,0,0) | (3,2,2) | (1,2,2) | ← 从 P1 开始 |
| P2 | (3,0,2) | (9,0,2) | (6,0,0) | |
| P3 | (2,1,1) | (2,2,2) | (0,1,1) | |
| P4 | (0,0,2) | (4,3,3) | (4,3,1) | |

安全序列：`<P1, P3, P4, P2, P0>` ✓

**实测**：
```
[banker] init OK (nres=3, available=(3,3,2))
[banker] initial safe sequence: <P1, P3, P4, P0, P2>
Test: P0 request (0,2,0) -> GRANTED (safe)
Test: P0 request (3,3,2) -> REFUSED (unsafe)
=== Banker Test PASSED ===
```

**占位符说明**：
- [图片20] bankertest 运行截图
- [图片21] 安全性算法工作向量变化图

**讲解词**：
> 死锁避免的经典算法是银行家算法。核心思想是：每次资源分配前，模拟分配并运行安全性检查，确保存在安全序列才真正分配。左表是教科书上的 T0 状态，我们的安全序列 `<P1, P3, P4, P2, P0>` 与教材完全一致。实测显示，P0 请求 (0,2,0) 被允许，但请求 (3,3,2) 被拒绝，因为会进入不安全状态。

### 第 14 页 — 优先级继承（Mars Pathfinder）

**标题**：经典问题：优先级反转与优先级继承

**内容**：

**问题场景**（1997 火星探测器）：
```
L (low)  ─── 持锁 ──▶  M (medium) ─── busy-loop ──▶  H (high) 等待锁
                             
无 PI: M 抢占 L → L 无法释放锁 → H 永远等待 → 系统 hang
有 PI: L 继承 H 优先级 → M 无法抢占 → L 完成 → H 继续
```

**实现**：
```c
// sem_wait 中
if (holder) {
  holder->priority = min(holder->priority, my_priority);
  holder->boost_count++;
}

// sem_post 中
if (p->boost_count > 0) {
  p->boost_count--;
  if (p->boost_count == 0)
    p->priority = p->orig_priority;  // 恢复
}
```

**实测**：
```
[L  pid=4] acquired sem, doing critical work
[M  pid=5] busy-looping
[L  pid=4] releasing sem, prio=0    ← L 被提升到 H 优先级
[H  pid=6] got sem after 7028888 us
=== Phase D1 PASSED ===
```

**占位符说明**：
- [图片24] pathfinder 程序运行截图
- [图片25] 优先级反转时间线图

**讲解词**：
> 这是 1997 年 NASA 火星探测器的真实 bug。低优先级进程 L 持有锁，中优先级 M 在 busy-loop，高优先级 H 等待 L 持的锁。无优先级继承时，M 抢占 L，L 无法释放锁，H 永远等待——系统看似 hang。我们的实现中，H 阻塞时 L 临时继承 H 的优先级，M 无法抢占，L 完成后 H 在约 7ms 内获得资源，系统正常运行。

### 第 15 页 — 实时调度（RM/EDF）

**标题**：实时调度：Rate-Monotonic 与 EDF

**内容**：

**Rate-Monotonic（RM）**：
```
优先级 = 1 / 周期（周期越短优先级越高）
可调度条件: Σ(Ci/Ti) ≤ n(2^(1/n) - 1)
```

**测试配置**（利用率 0.60 < 0.779）：

| 任务 | 周期 Ti | 执行 Ci | 利用率 |
|------|---------|---------|--------|
| T1 | 10 | 2 | 0.20 |
| T2 | 20 | 4 | 0.20 |
| T3 | 40 | 8 | 0.20 |

**EDF（最早截止时间优先）**：
```
优先级 = 1 / 截止时间（截止时间越近优先级越高）
可调度条件: Σ(Ci/Ti) ≤ 1（理论可达 100% 利用率）
```

**对比**：

| 维度 | RM | EDF |
|------|-----|-----|
| 优先级类型 | 静态 | 动态 |
| 可调度上限 | ≈ 69% | 100% |
| 实现复杂度 | 低 | 中 |

**占位符说明**：
- [图片26] rmtest 运行截图
- [图片27] RM vs EDF 调度时间线对比图

**讲解词**：
> 实时调度是高级调度扩展。Rate-Monotonic 是静态优先级调度，周期越短优先级越高，可调度上限约 69%。我们的测试配置利用率 0.60，通过理论验证。EDF 是动态优先级调度，按截止时间分配优先级，理论上可达 100% 利用率但实现更复杂。两种方法各有适用场景。
