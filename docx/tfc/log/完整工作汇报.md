### 第 1 页 — PCB 数据结构图（F1）

**功能示例**：实现进程控制块（PCB）数据结构

**代码来源**：`kernel/proc.h` 第 97-157 行 `struct proc`

**工作脉络**：要管理进程，首先得有数据结构来描述进程。xv6 原版 PCB 只有 state、lock、chan、pagetable 等基础字段，远远不够。我们的 PCB 扩展了三大类共 **20+ 个字段**，为后续所有功能打好基础。

**讲解词（约 50 秒）**：

> 首先看进程控制块 PCB——这是内核最重要的数据结构。**为什么要先讲 PCB？** 因为操作系统要管理进程，必须先把"进程是什么"定义清楚。xv6 原版 PCB 字段很少，我们的 PCB 扩展了三大类共 **20+ 个字段**，图里用不同背景色区分。
>
> **浅灰底是调度相关**：这些字段支撑了后面 6 种调度算法——queue_level 支撑 MLFQ 五级队列；priority 支撑优先级调度；est_burst 支撑 SJF；rt_period/rt_deadline 支撑 EDF 实时调度；cpu_affinity 支撑 CPU 亲和性。可以说，**没有这些字段，后面的调度器就是无根之木**。
>
> **横线底是同步与等待**：waitbucket 指针把进程链入 wait 哈希桶（后面第 2 页会展开）；child_count 配合 wait_lock 保护父子关系。
>
> **点阵底是统计**：ctime、wait_time、run_time、sched_count——这些字段记录了进程的一生，死锁检测器也用它们判断是否"假死锁"。
>
> 答辩时被问"PCB 字段的作用"，直接指着对应颜色说：基础字段是 xv6 原版，**调度字段是我加的，同步字段是我加的，统计字段也是我加的**，加起来 20+ 个。

**逻辑图**（见 [F1](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F1)）：

- 左列：基础字段（state / lock / chan / pid / kstack / sz）
- 中列：调度字段（queue_level / priority / est_burst / rt_period / cpu_affinity）
- 右列：同步与统计字段（waitbucket / child_count / ctime / wait_time / run_time）
- 底部：PCB 在 proc[NPROC] 全局数组中的位置

**文字说明**：

xv6 原版 PCB 字段很少。我们的扩展 PCB 分为三类：

| 类别 | 字段 | 用途 |
|------|------|------|
| 基础 | state, lock, chan, killed, xstate, pid, kstack, sz, pagetable, trapframe, context | 进程基本信息和运行环境 |
| 调度 | queue_level(0-4), priority(0-10), est_burst, rt_period, rt_cost, rt_deadline, rt_release, cpu_affinity | 支撑 6 种调度算法 |

### 第 2 页 — 进程状态机与 Wait 哈希桶图（F2）

**功能示例**：实现进程状态转换 + wait 哈希桶优化

**代码来源**：`kernel/proc.h` 第 31-34 行 `struct waitbucket`；`kernel/proc.c` 第 13 行 `waittable[64]`；第 149-180 行

**工作脉络**：基本进程生命周期走通后，开始做性能优化。发现 xv6 的 wakeup() 对每个等待通道都要遍历整个 proc[NPROC] 表（64 项），**这是第一个性能瓶颈**。于是引入 Wait 哈希桶优化。

**发现问题**：原始 xv6 的 `wakeup(chan)` 对每个等待通道都要遍历整个 `proc[NPROC]` 表（64 项）。当大量进程分别在不同通道上睡眠时（如 64 个进程各自等待自己的 `&p`），每次 `wakeup` 都是 O(64) 扫描。

**解决方案**：64 桶哈希表 `waittable[NWCHAN]`，按 wait channel 散列：

```c
// proc.c:13 — 全局哈希桶数组
struct waitbucket waittable[NWCHAN];  // NWCHAN = 64

// proc.c:149-153 — 哈希函数
static struct waitbucket *waitbucket_for(void *chan) {
  return &waittable[((uint64)chan) % NWCHAN];
}
```

**讲解词（约 50 秒）**：

> 这张图有两部分——左侧是进程状态机，右侧是我们做的第一个优化：**wait 哈希桶**。
>
> **为什么要优化 wait？** 在 xv6 原版里，`wakeup(chan)` 要遍历整个 proc 表（64 项）才能找到等待该 channel 的进程。**这就好比图书馆里找人，要一间一间地翻——能不能建个索引？** 于是我们引入了 64 个哈希桶，把进程按 wait channel 散列到不同桶。
>
> **左侧 6 态转换**：UNUSED→USED→RUNNABLE→RUNNING→SLEEPING→ZOMBIE，6 种状态 8 条转换路径，和教材完全一致。
>
> **右侧是哈希桶机制**：每个桶是一张链表，`waitbucket_for(chan) = chan % 64`。wakeup 时只要遍历对应桶的链表即可。**效果**：最坏 O(N)，平均 O(N/64)，64 进程下从扫描 64 个降到 1-2 个。
>
> 答辩时被问"wait 优化效果"，直接说：平均复杂度降了一个数量级。**这是我们在 xv6 原版基础上做的第一个 kernel 层面的性能优化。**

**逻辑图**（见 [F2](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F2)）：

- 左：6 态状态机（UNUSED / USED / RUNNABLE / RUNNING / SLEEPING / ZOMBIE）
- 中：`waittable[64]` 哈希表，hash = chan % 64
- 右：单个桶的链表结构（proc -> wnext -> wnext -> null）

**文字说明**：

| 操作 | 原始 xv6 | 优化后 |
|------|---------|--------|
| wakeup(chan) | O(NPROC) 全表扫描 | O(waiters_on_channel) |
| 锁竞争 | 全局 wait_lock | per-bucket lock |
| 典型场景 | 64进程各等各的 → 每次扫描64 | 每次扫描1 |

### 第 3 页 — fork / exec / exit / wait 完整流程图（F3）

**功能示例**：实现进程创建（fork/exec）+ 退出与等待

**代码来源**：`kernel/proc.c` 第 405-484 行 `kfork()`；`kernel/exec.c` 第 27-151 行 `kexec()`；`kernel/proc.c` 第 528-567 行 `kexit()`；第 571-616 行 `kwait()`

**工作脉络**：有了 PCB 和状态机，现在把 fork→exec→exit/wait 的完整链路走通。这是所有操作系统最核心的进程管理逻辑——**不会 fork/exec，系统跑不起来任何程序**。

**讲解词（约 60 秒）**：

> 这张图把进程的一生（fork→exec→运行→exit/wait）画成一张完整泳道图，左边用户态、右边内核态，四条流程纵向展开。**为什么要讲这四个流程？因为这是 xv6 进程管理的四大基本操作——进程从无到有、换程序、退出、被回收，全在这四条链路里。**
>
> **第一个流程 fork**（顶部）：`kfork()` 做了 8 件事——allocproc 分配 PCB、uvmcopy 复制用户内存、复制 trapframe、复制文件描述符、设置 parent 链表、设为 RUNNABLE。**注意 uvmcopy 是全量复制，不是 COW**，每次 fork 代价约 450μs。另外，fork 时会复制共享内存映射（proc.c:425-447），这是后续 IPC 的基础。
>
> **第二个流程 exec**（中上）：`kexec()` 重新加载程序——setupkvm 创建空页表、遍历 ELF LOAD 段分配物理页并加载、分配用户栈和 guard page、最后**原子切换**（oldpagetable→newpagetable）。这一步约 1200μs。
>
> **第三个流程 exit+reparent**（中下）：`kexit()` 关闭文件、释放 inode、调用 `reparent()` 将所有子进程转交给 initproc，然后变为 ZOMBIE。
>
> **第四个流程 wait**（底部）：`kwait()` 先在 wait 哈希桶里找 ZOMBIE 子进程，找到后调用 `freeproc()` 释放资源。**找不到则 sleep**，等待 exit 唤醒。
>
> **调试花絮**：我们在调试 reparent() 时发现 `cnext` 字段在 fork 中被当作"兄弟链表"使用，在 reparent 中却被当作"子进程链表"遍历——两个逻辑冲突了。修复：reparent() 改为全表扫描 `parent == p`。这类字段命名歧义的 bug 是 xv6 扩展时的常见陷阱。

**逻辑图**（见 [F3](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F3)）：

- 双泳道：用户态（fork/exec/wait/exit） / 内核态（kfork/kexec/kexit/kwait）
- 4 个流程纵向排列，每个流程标出关键函数调用和返回值

**文字说明**：

**kfork 的七步链路**（proc.c:405-484）：

1. `allocproc()` — 扫描 proc[NPROC] 找 UNUSED slot，分配 PID，设 trapframe，创建 fresh pagetable
2. `uvmcopy(p->pagetable, np->pagetable, p->sz)` — 全量复制用户内存
3. **复制 SHM 映射**（proc.c:425-447）— 检测 `SHM_BASE` PTE，同一物理页映射到子进程
4. `*(np->trapframe) = *(p->trapframe)`；`np->trapframe->a0 = 0` — 子进程 fork 返回值为 0
5. 复制 `ofile[NOFILE]`（filedup）；`idup()` cwd
6. 持有 `wait_lock` 将子进程加入父进程的子进程链表（cnext/cprev）
7. `np->state = RUNNABLE`，返回 child PID

**kexec 的原子提交**（exec.c:27-151）：

```
1. namei() 打开 ELF 文件
2. proc_pagetable() 创建新页表（TRAMPOLINE + TRAPFRAME）
3. loadseg() × N — 遍历 ELF LOAD 段，uvmalloc 分配物理页，readi 复制数据
4. uvmalloc(stack+guard) — 分配用户栈，含 guard page
5. copyout argc/argv 到栈
6. [原子提交] old = pagetable; pagetable = new; sz = sz; epc = elf.entry
7. proc_freepagetable(old) — 失败则回滚旧状态
```

### 第 4 页 — 上下文切换与调度器入口图（F4）

**功能示例**：实现进程调度器入口与上下文切换

**代码来源**：`kernel/swtch.S`；`kernel/proc.c` 第 725-753 行 `rr_scheduler()`；第 1213-1231 行 `sched()`；第 1234-1254 行 `yield()`

**工作脉络**：进程有了状态和生命周期，接下来要知道**进程怎么让出 CPU、调度器怎么选下一个**。这为第二阶段的调度器实现埋下伏笔——没有上下文切换，调度器无从谈起。

**讲解词（约 50 秒）**：

> 有了 PCB 和状态，下一个问题是：**CPU 怎么从一个进程切换到另一个进程？** 这张图展示了上下文切换的完整过程。
>
> **上半部分是 swtch 汇编**（`kernel/swtch.S`）：`swtch(old, new)` 把当前上下文（ra/sp/s0-s11）保存到 old，再从 new 加载。12 个寄存器 × 8 字节 = 96 字节，这决定了进程切换的最小开销。
>
> **下半部分是调度主入口**（`kernel/proc.c` scheduler()）：每个 CPU 的 `scheduler()` 是一个无限循环——关中断→扫描 proc 表→选一个 RUNNABLE 进程→swtch→切回调度器→开中断→再循环。
>
> **图里重点标红的是 `wfi` 指令**（`kernel/proc.c` 第 751 行）——当 proc 表里没有任何 RUNNABLE 进程时，CPU 执行 `wfi`（Wait For Interrupt）进入低功耗等待，被时钟中断唤醒后继续扫描。**这样设计，CPU 在没有进程可跑时完全不耗电。**
>
> **注意**：当前配置下只有 hart 0 运行 scheduler，但代码本身是 per-CPU 的。**这就是为什么 F1 的 PCB 里要有 cpu_affinity 字段——为多核做准备。**

**逻辑图**（见 [F4](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F4)）：

- 顶部：swtch.S 寄存器保存/恢复示意图（12 个寄存器框）
- 中部：scheduler() 主循环伪代码（while(1) { intr_off; scan; pick; swtch; intr_on }）
- 底部：wfi 低功耗等待 + 时钟中断唤醒的循环

**文字说明**：

**swtch.S 上下文切换**：

```asm
swtch:
  sd ra, 0(a0); sd sp, 8(a0); sd s0, 16(a0); ...
  ld ra, 0(a1); ld sp, 8(a1); ld s0, 16(a1); ...
  ret
```

- 保存 12 个寄存器（ra + sp + s0-s11）到 old context
- 从 new context 恢复 12 个寄存器
- `ret` 跳转到 new context 的 ra（即 new 进程的下一条指令）

**scheduler() 主循环**：

```c
// proc.c:682-703
void scheduler(void) {
  for(;;) {
    intr_off();
    if(current_scheduler == SCHED_FCFS)    fcfs_scheduler();
    else if(current_scheduler == SCHED_MLFQ) mlfq_scheduler();
    else if(current_scheduler == SCHED_SJF)  sjf_scheduler();
    else if(current_scheduler == SCHED_PRIO) prio_scheduler();
    else if(current_scheduler == SCHED_EDF)  edf_scheduler();
    else                                    rr_scheduler();
    intr_on();
  }
}
```

### 第 5 页 — 共享内存与消息队列 IPC 图（F5）

**功能示例**：实现进程间通信（共享内存 + 消息队列）

**代码来源**：`kernel/proc.c` kfork() 第 425-447 行（共享内存映射复制）；`kernel/shm.c`；`kernel/msgq.c`

**工作脉络**：进程生命周期走通后，自然想到：**进程之间怎么传递数据？** xv6 原版只有 pipe，我们扩展了共享内存和消息队列。同时发现 fork 时共享内存映射要一起复制过去——这需求是在做 fork 时自然发现的，所以 IPC 放在第一阶段最后讲。

**讲解词（约 40 秒）**：

> 前面讲了进程怎么创建、怎么切换，但**多个进程之间怎么通信？** xv6 原版只有 pipe，我们扩展了两种更高效的 IPC。
>
> **左半是共享内存**：`shmget()` 在物理内存中分配共享页，`shmat()` 将其映射到进程虚拟地址空间。**关键设计**：fork 时内核自动复制共享内存映射（proc.c:425-447），父子进程直接读写同一块物理页，**零拷贝**，适合大数据交换。调试时发现 `sys_shmat()` 误把用户态指针地址传给了内核函数，导致 kerneltrap——修复后正常。
>
> **右半是消息队列**：`msgget()` 创建队列，`msgsnd()`/`msgrcv()` 走内核缓冲区复制数据。这是有界缓冲区的生产者-消费者模式——**注意它不会死锁**，因为消息队列有容量上限，满则阻塞。
>
> **IPC 和同步的关系**：这里的消息队列用 sleep/wakeup 实现同步（队列满/空），这正是第三阶段信号量的雏形——**IPC 和同步天然是不可分割的**。

**逻辑图**（见 [F5](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F5)）：

- 左：shmget → shmat → 父子进程共享同一物理页（虚线表示映射）
- 右：msgqget → msgsnd → 内核队列 → msgrcv（生产者-消费者模式）

**文字说明**：

**共享内存实现**（shm.c）：

| 常量/结构 | 值/字段 |
|-----------|--------|
| NSHM | 16（最大段数） |
| SHM_BASE | 0x3FEFE000（固定映射地址） |
| SHM_SIZE | 0x1000（每段 4KB） |

```c
// kernel/shm.h
struct shm {
  char name[32];
  uint64 phys_addr;    // 物理页地址
  int refcount;        // 引用计数（shmat + fork）
  int allocated;
  int key;
};
```

**fork 时复制 SHM 映射**（proc.c:425-447）：检测 `SHM_BASE` PTE，同一物理页映射到子进程，`refcount++`。

**调试发现**（Done.md §共享内存 sys_shmat panic）：`sys_shmat()` 误把用户态指针地址传给 `shmat()`，导致 copyout 写入用户地址时触发 kerneltrap。修复：区分用户态指针和内核结果，用 `copyout()` 正确回传。

**消息队列实现**（msgq.c）：

| 常量/结构 | 值/字段 |
|-----------|--------|
| NMSGQ | 16（最大队列数） |
| NMSG | 8（每队列最大消息数） |
| MAX_MSG_SIZE | 128（每消息最大字节） |

- `msgsnd()` — 队列满时 `sleep(notfull, lock)`；发送后 `wakeup(notempty)`
- `msgrcv()` — 队列空时 `sleep(notempty, lock)`；接收后 `wakeup(notfull)`

## 第二阶段：智能化调度——6 种调度算法

> **阶段背景**：第一阶段走通后，自然想到：**谁来决定让哪个进程跑？** xv6 原版只有一种简单 RR 调度器——所有进程排成一队，轮流转。**问题**：这种"一视同仁"的策略太笨——不能区分 I/O 密集型（频繁阻塞）和 CPU 密集型（一直占 CPU），短作业响应差。**于是**：我们实现了 6 种调度算法，通过 `sched_algorithm()` 系统调用运行时切换，无需重启内核。

### 第 6 页 — 6 种调度器总览架构图（F6）

**功能示例**：实现进程调度器（支持 6 种算法运行时切换）

**代码来源**：`kernel/proc.c` 第 682-703 行 `scheduler()`；`kernel/sysproc.c` 第 242-272 行 `sys_sched_algorithm()`

**工作脉络**：第一阶段发现"轮流转"策略的问题。**设计决策**：不维护 6 套独立的 scheduler，而是用统一入口 + 分发模式，这样代码量最少且易于维护。

**讲解词（约 55 秒）**：

> 第二阶段来了。**第一阶段解决了"进程怎么跑起来"，第二阶段要回答"谁来决定让谁跑"**——这就是调度器。
>
> **顶部是统一入口** `scheduler()`（第 682-703 行）——它本身不选进程，而是读 `current_scheduler` 全局变量，分发到对应的调度函数。**这种委托模式是核心设计**，所以我们能在同一个内核里跑 6 种算法。**为什么不写 6 个独立的 scheduler？因为共用 proc 表遍历、共用 swtch 逻辑，代码复用最省事。**
>
> **下方 6 个调度器并列**：RR / FCFS / MLFQ / SJF / PRIO / EDF，用 6 种填充模式区分。默认是 **MLFQ**（细斜线）——这是 MLFQ 被选为默认的理由，我们会在 F10 展开讲。
>
> **右侧是运行时切换机制**：`sched_algorithm(n)` 系统调用（syscall #38）接收 0-5，直接改 `current_scheduler`，然后调用 `yield()` 强制重新调度。**无需重新编译内核**，这是和 xv6 原版最大的区别。
>
> 调试时有个花絮：最开始 scheduler() 里把 algo 缓存到局部变量，结果 `sched_algorithm()` 切了算法但调度器还在用旧值——修复方法是每轮重新读全局变量。**这是一个经典的"全局变量缓存失效"陷阱。**

**逻辑图**（见 [F6](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F6)）：

- 顶部：scheduler() 主循环（读 current_scheduler → 分发）
- 下方 6 格：RR / FCFS / MLFQ / SJF / PRIO / EDF，各有 pick_next 伪代码片段
- 右侧：sched_algorithm() 系统调用 + 时间片配置表

**文字说明**：

```c
// proc.c:682-703 — 调度分发器
void scheduler(void) {
  for(;;) {
    intr_off();
    if(current_scheduler == SCHED_FCFS)    fcfs_scheduler();
    else if(current_scheduler == SCHED_MLFQ) mlfq_scheduler();
    else if(current_scheduler == SCHED_SJF)  sjf_scheduler();
    else if(current_scheduler == SCHED_PRIO) prio_scheduler();
    else if(current_scheduler == SCHED_EDF)  edf_scheduler();
    else                                    rr_scheduler();
    intr_on();
  }
}
```

**运行时切换机制**：

```c
// sysproc.c:242-272
uint64 sys_sched_algorithm(void) {
  int algo;
  argint(0, &algo);
  if(algo == -1) return current_scheduler;
  acquire(&sched_lock);
  int old = current_scheduler;
  current_scheduler = algo;
  release(&sched_lock);
  if(algo != old && myproc() != 0)
    yield();  // 立即 yield，让调度器用新算法重选
  return old;
}
```

**六种算法选取规则**：

| 算法 ID | 算法名 | pick_next 规则 | 抢占性 |
|---------|--------|---------------|--------|
| 0 | RR | first RUNNABLE in scan | tick yield |
| 1 | FCFS | min ctime | 非抢占 |
| 2 | MLFQ | min queue_level, then min ctime | tick 降级 |
| 3 | SJF | min est_burst, then min ctime | 非抢占 |
| 4 | PRIO | min priority, then min last_sched | tick aging |
| 5 | EDF | min rt_deadline | 抢占 |

### 第 7 页 — MLFQ 五级队列与 Aging 图（F7）

**功能示例**：实现高级调度（MLFQ 多级反馈队列）

**代码来源**：`kernel/proc.c` 第 810-829 行 `mlfq_boost_priority()`；第 831-891 行 `mlfq_scheduler()`；`kernel/trap.c` 第 82-103 行

**工作脉络**：RR 的问题是所有进程一视同仁，短作业要等很久。**解决方案**：多级反馈队列——给交互式短作业高优先级，给 CPU 密集型长作业低优先级。同时要解决"饥饿"问题：低优先级进程永远得不到 CPU → 引入 Aging 提升。

**讲解词（约 55 秒）**：

> 6 种调度器里最复杂也最实用的是 **MLFQ（多级反馈队列）**——它是 RR 的升级版，解决了 RR 的核心痛点：**所有进程一视同仁，短作业要等很久**。
>
> **左侧是 5 级队列**：Q0（最高）时间片 1 tick、Q1 是 2 tick、Q2 是 4 tick、Q3 是 8 tick、Q4（最低）是 15 tick。**填充密度递增**（白→细斜线→粗斜线），视觉上体现优先级递减。**这个设计思想是**：交互式短作业用完 1 tick 就阻塞 I/O，留在 Q0；CPU 密集型长作业一直占满时间片，被降到 Q4。
>
> **两条关键箭头**：**实线红箭头向下是降级**——进程用完时间片就降一级（trap.c:87-102 在时钟中断里执行）；**虚线绿箭头向上是提升**——每 100 ticks 的 `mlfq_boost_priority()` 把所有进程拉回 Q0。**降级防止长作业占高优，提升防止低优先级进程饥饿**——两个机制缺一不可。
>
> **I/O 保护**（答辩亮点）：wakeup() 重置 `timeslice_used = 0`，睡眠进程不降级——I/O 密集型进程被唤醒后保持在原队列，不因阻塞而受惩罚。
>
> **调试花絮**：测 SJF 时发现子进程 fork 后在用户态调用 `sched_setburst()`，但调度器选择发生在子进程被选中之前——burst 字段对调度器"不可见"。解决：用 sem barrier 让所有子进程先注册再切 SJF。**这说明用户态和内核态的时序协调是一个常见陷阱。**

**逻辑图**（见 [F7](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F7)）：

- 5 个队列横排：Q0(1 tick) / Q1(2 tick) / Q2(4 tick) / Q3(8 tick) / Q4(15 tick)
- 降级箭头（实线红）：用完时间片 → 降一级
- 提升箭头（虚线绿）：每 100 ticks → 全部回 Q0
- 右侧：3 个进程的轨迹示意

**文字说明**：

**MLFQ 时间片配置**（param.h）：


**I/O 保护**：wakeup() 重置 `timeslice_used = 0`，睡眠进程不降级——I/O 密集型进程被唤醒后保持在原队列，不因阻塞而受惩罚。

### 第 8 页 — 实时调度：RM + EDF 图（F8）

**功能示例**：实现实时调度（Rate-Monotonic + EDF）

**代码来源**：`kernel/proc.c` 第 1078-1155 行 `edf_scheduler()`；`kernel/sysproc.c` 第 326-356 行 `rt_register()`

**工作脉络**：MLFQ 解决通用场景，但实时系统有更严格的需求——**每个任务必须在截止期前完成**。于是我们实现了 RM 和 EDF 两种实时调度算法。

**讲解词（约 50 秒）**：

> MLFQ 解决了通用场景的"短作业响应"问题，但实时系统有更严格的需求：**每个任务必须在截止期前完成**，完不成就是事故。**这就是实时调度的由来。**
>
> **左侧是 Rate-Monotonic（RM）调度**：它是固定优先级调度（PRIO 调度的特例），周期越短优先级越高。图中三个任务 H/M/L 周期分别是 4/8/16 tick，CPU 利用率 0.75。**RM 的可调度条件**是：总利用率 U ≤ n(2^(1/n) - 1)，n=3 时 ≈ 0.78，0.75 < 0.78，所以全部满足 deadline。
>
> **右侧是 EDF（Earliest Deadline First）**：动态优先级调度，每次选 **rt_deadline 最近的进程**。图中三个同周期任务的 deadline 不同——先 fork 的进程 release 时间早，deadline 也最早，所以被优先调度。
>
> **下方是实时任务的 PCB 字段**：rt_period（周期）、rt_cost（每周期最大 CPU 时间）、rt_deadline（绝对截止期）。这些字段通过 `rt_register(period, cost)` 系统调用设置。
>
> **RM vs EDF 的选择**：EDF 是最优动态优先级算法——如果存在某个调度序列满足所有 deadline，EDF 一定能找到。但 RM 实现更简单，且实时系统常要求确定性（可预测）——两者各有适用场景。

**逻辑图**（见 [F8](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F8)）：

- 左：RM 调度时序图（H period=4 / M period=8 / L period=16）
- 右：EDF 调度 Gantt 图（deadline 最近优先）
- 下方：实时任务 PCB 字段表

**文字说明**：

**实时 PCB 字段**（proc.h:136-141）：

```c

```

**RM vs EDF**：

| 特性 | RM | EDF |
|------|-----|-----|
| 优先级类型 | 固定（静态） | 动态 |
| 可调度条件 | U ≤ n(2^(1/n) - 1) | U ≤ 1（充分条件） |
| 最优性 | 最优静态优先级 | 最优动态优先级 |

### 第 11 页 — 信号量 P/V + 优先级继承图（F11）

**功能示例**：实现进程间同步机制：信号量 + 优先级继承

**代码来源**：`kernel/sem.c` 第 70-126 行 `sem_wait()`；第 129-170 行 `sem_post()`

**工作脉络**：从调度器进入同步领域，第一步是实现**信号量**——最基础的同步原语。xv6 原版没有信号量，我们从零实现。实现过程中发现 Mars Pathfinder 优先级反转问题，于是加入优先级继承（Priority Inheritance）修复。

**讲解词（约 55 秒）**：

> 第二阶段解决了"谁上 CPU"，第三阶段要回答"多个进程怎么安全地配合"。**这是从"单兵作战"到"团队协作"的跨越**——也是操作系统最核心的挑战之一。
>
> 首先实现**信号量**——最基础的同步原语。xv6 原版没有信号量，我们从零实现。**信号量是什么？** 它是一个整数 + 一个等待队列。P 操作（sem_wait）把 count 减一，如果小于零就睡眠；V 操作（sem_post）把 count 加一，如果小于等于零就唤醒一个等待者。
>
> **但在实现过程中，我们发现了 Mars Pathfinder 经典问题**：低优先级进程 L 持有信号量，高优先级进程 H 等待该信号量，中优先级进程 M 抢占 L 的 CPU 时间，导致 L 无法释放锁，H 永久等待——**系统崩溃**。
>
> **解决方案是优先级继承（PI）**：当 H 等待 L 持有的信号量时，**把 L 的优先级临时提升到 H 的级别**。这样 M 无法抢占 L，L 快速完成并释放锁，H 被唤醒。**我们在 sem_wait 中检测这种情况并 boost，在 sem_post 中恢复**——调试时发现 PI 恢复要放在 if/else 外，否则无 waiter 时不恢复，持锁进程永久升高。
>
> **注意**：信号量和 wait 哈希桶的 sleep/wakeup 是同一个底层机制——wait 哈希桶是 sleep/wakeup 的优化，信号量建立在 sleep/wakeup 之上。

**逻辑图**（见 [F11](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F11)）：

- 左：P 操作流程（value-- → <0? → sleep : set holder_pid）
- 右：V 操作流程（value++ → ≤0? → wakeup : clear holder_pid）
- 底部：优先级继承时序（L 持有 sem → H 等待 → L priority boost → M 无法抢占 → H 被唤醒）

**文字说明**：

**信号量数据结构**（sem.h）：

```c
#define NSEM 16
struct semaphore {
  struct spinlock lock;
  int value;
  int allocated;
  char name[16];
  struct sem_waiter *waiters;
  int holder_pid;         // 当前持有者 PID（PI 用）
};
```

**P 操作（sem_wait）**：

```c
// sem.c:70-126
void sem_wait(int sem_id) {
  struct semaphore *sem = &semtable[sem_id];
  acquire(&sem->lock);
  sem->value--;
  if(sem->value < 0) {
    sleep(sem, &sem->lock);
  } else {
    sem->holder_pid = myproc()->pid;
  }
  release(&sem->lock);
}
```

**优先级继承 — Mars Pathfinder Fix**：

```c
// sem.c:86-110 — 在 sem_wait 中
if(sem->value < 1 && sem->holder_pid >= 0 && sem->holder_pid != my_pid) {
  struct proc *holder = find_by_pid(sem->holder_pid);
  if(holder && my_prio < holder->priority) {
    if(holder->boost_count == 0)
      holder->orig_priority = holder->priority;
    holder->priority = my_prio;  // 提升到请求者级别
    holder->boost_count++;
  }
}
```

**调试发现**（Done.md §D1 关键发现与修复）：原本 `sem_post` 中 PI 恢复放在 `if(sem->value <= 0)` 内——无 waiter 时不恢复，导致持锁进程永远停留在高优先级。修复：移到 if/else 外，无论有无 waiter 都恢复。

### 第 12 页 — 管程 Mesa 风格实现图（F12）

**功能示例**：实现进程间同步机制：互斥锁（管程）+ Mesa 风格

**代码来源**：`kernel/monitor.c` 第 44-51 行 `struct monitor`；第 118-170 行

**工作脉络**：信号量解决了 P/V 操作，但**更高层的同步抽象**需要把共享数据和操作打包成一个"类"——这就是管程（Monitor）。信号量可以互相嵌套，容易死锁；管程天然互斥，更易读。

**讲解词（约 50 秒）**：

> 信号量虽然强大，但有一个问题：**P/V 操作可以嵌套**，多个信号量操作交织在一起容易形成死锁。**怎么解决这个问题？** 管程（Monitor）提供了一个更高层的抽象——把共享数据和操作打包成一个"类"，**内部天然互斥**，使用者无需操心锁的获取和释放。
>
> **左侧是管程结构**：每个 monitor 占用 4 个连续信号量（1 个 mutex 初值=1 + 3 个条件变量初值=0），最多同时存在 8 个 monitor。
>
> **右侧是 Mesa vs Hoare 对比**：Hoare 风格（signal 后立即切换给 waiter）语义更清晰但实现复杂；**Mesa 风格（signal 后当前进程继续运行）**实现简单，但 waiter 被唤醒后必须 `while(cond)` 重新检查条件——**为什么？因为唤醒后条件可能已被其他进程改变了**。xv6 选 Mesa，因为 wait 队列天然就是 Mesa 的链表结构。
>
> **底部是生产者-消费者模板**：用 monitor 实现比用信号量实现简洁得多——`monitor_lock()` → `monitor_wait()` → 生产 → `monitor_signal()` → `monitor_unlock()`，逻辑一目了然。**这说明管程和信号量的关系**：管程是用信号量实现的，信号量是底层原语，管程是高层抽象。

**逻辑图**（见 [F12](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F12)）：

- 左：monitor 内部结构（mutex + 3 个 cv + 共享数据）
- 中：Mesa 风格时序（signal 后当前进程继续，waiter 重新 while 检查）
- 右：Hoare 风格时序（signal 后立即切换给 waiter）

**文字说明**：

**Mesa 语义实现**：

```c
// monitor_wait — Mesa 风格
void monitor_wait(int mid, int cvid) {
  sem_post(monitor[mid].mutex);       // 释放互斥锁
  sem_wait(monitor[mid].cv[cvid]);     // 睡在条件变量上
  sem_wait(monitor[mid].mutex);        // 被唤醒后重新竞争互斥锁
}
```

| 特性 | Mesa | Hoare |
|------|------|-------|
| signal 语义 | signal-and-continue | signal-and-run |
| 条件检查 | 必须 while(cond) | if(cond) 就够了 |
| 实现复杂度 | 简单 | 复杂 |

### 第 13 页 — 死锁检测：Wait-For Graph + DFS 图（F13）

**功能示例**：实现死锁检测与恢复

**代码来源**：`kernel/deadlock_detect.c` 第 84-282 行

**工作脉络**：信号量和管程解决了"怎么安全地配合"，但**程序员写错了还是会死锁**。于是引入死锁检测器——每 30 ticks 扫描 Wait-For Graph，发现环就 abort 最年轻的进程。

**讲解词（约 55 秒）**：

> 信号量和管程解决了"怎么安全地配合"，但**程序员的错误是防不住的**——如果代码写错了形成了循环等待怎么办？**这就引出了死锁检测器**。
>
> **左侧是 Wait-For Graph（等待图）**：每个节点是一个进程，每条边 P_i→P_j 表示"P_i 在等待 P_j 持有的资源"。当图中出现**环**时，就是死锁——**这正是 F15 哲学家就餐问题的数学抽象**。
>
> **右侧是内核实现**：每 30 ticks 的 `deadlock_scan()` 做三件事——**①建图**：扫描所有 SLEEPING 进程，根据它们的 `p->chan`（等待的信号量）建立有向边；**②检测环**：用 DFS 遍历图，找是否有回路；**③恢复**：选环中 **PID 最大的进程（最年轻）作为 victim**，调用 `abort_proc()` 回收它的锁并发送 kill 信号。
>
> **底部是假死锁排除**：如果系统中存在任何 RUNNABLE 或 RUNNING 进程，就说明系统有进展，**不触发检测**。这避免了把正常的生产者-消费者阻塞误判为死锁。**我们的检测器还做了自适应扩展**：扫描间隔动态调整（5-120 tick）、多维 victim 评分、held_sem_ids 精确追踪——这是我们超越教科书方案的地方。

**逻辑图**（见 [F13](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F13)）：

- 左：Wait-For Graph（进程节点 + 有向边）
- 中：deadlock_scan() 伪代码（建图 → DFS → abort）
- 右：假死锁排除逻辑（RUNNABLE 存在 → 不触发）

**文字说明**：

**DFS 环检测**（deadlock_detect.c:150-165）：

```c
int dfs_cycle(int u) {
  if(visiting[u]) return 1;  // 回边 = 环
  if(visited[u])  return 0;
  visiting[u] = 1;
  for(v : edges_from[u]) if(dfs_cycle(v)) return 1;
  visiting[u] = 0; visited[u] = 1;
  return 0;
}
```

**防御性设计**：

| 机制 | 描述 |
|------|------|
| 进展启发 | 有 RUNNABLE/RUNNING 进程 → 跳过检测 |
| 持久性守卫 | 死锁模式持续 DEADLOCK_PERSIST_TICKS=90 才触发 |
| victim 选择 | PID 最大（最年轻），Silberschatz §7.6 标准做法 |

**自适应死锁检测器创新点**（Done.md §B4）：自适应扫描间隔（5-120 tick）、多维 victim 评分、held_sem_ids 精确追踪、死锁历史日志（16 条环形缓冲区）。

### 第 14 页 — 银行家算法安全检查图（F14）

**功能示例**：实现银行家算法（死锁避免）

**代码来源**：`kernel/banker.c` 第 44-80 行 is_safe()；第 139-177 行 banker_request()

**工作脉络**：死锁检测是"发生了再处理"，银行家算法是"请求前先预测"——**预防优于检测**。如果分配后系统会进入不安全状态，就**拒绝请求**，永远不进入死锁。

**讲解词（约 50 秒）**：

> 死锁检测是"发生了再处理"，银行家算法是"请求前先预测"——**预防优于检测**。它的核心思想是：**每次资源分配前，先假装分配，看看系统是否还安全。如果不安全，就拒绝分配。**
>
> **左侧是数据结构**：Banker 维护 3 张表——`available[j]` 可用资源、`max[i][j]` 各进程最大需求、`allocation[i][j]` 当前分配。`need[i][j] = max[i][j] - allocation[i][j]`。
>
> **右侧是安全算法流程**：从当前 available 开始，贪心地找 need ≤ available 的进程，加 its allocation 到 available，标记 finish，重复直到所有 finish 或找不到。如果所有 finish=True → 安全。Done.md 实测经典 5 进程/3 资源案例，找安全序列 `<P1, P3, P4, P0, P2>`。
>
> **底部是 request 流程**：三步检查——① request ≤ need？② request ≤ available？③ 假装分配后调用 is_safe()？三步都通过才真正分配，否则拒绝。**这保证永远不会进入死锁状态。**
>
> **和死锁检测的对比**：死锁检测是事后诸葛亮，银行家是事前预防。**代价是需要预先知道每个进程的最大需求**——这在某些场景下不可行。F13 的死锁检测器不需要这个前提。

**逻辑图**（见 [F14](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F14)）：

- 左：Banker 数据结构（available / max / allocation / need 四表）
- 中：is_safe() 流程（work=available → 找 need≤work → finish → work+=allocation）
- 右：request 流程（request≤need? → request≤available? → is_safe? → GRANT/REFUSE）

**文字说明**：

**is_safe() 安全算法**（banker.c:44-80）：

```c
int is_safe(int out_seq[]) {
  int work[NRES], finish[NPROC_B];
  for(j) work[j] = available[j];
  for(i) finish[i] = 0;

  int found = 1, count = 0;
  while(found) {
    found = 0;
    for(i=0; i<nproc; i++) if(!finish[i]) {
      for(j) if(need[i][j] > work[j]) break;
      if(j == nres) {  // 可以完成
        for(j) work[j] += allocation[i][j];
        finish[i] = 1; out_seq[count++] = i; found = 1;
      }
    }
  }
  return (count == nproc) ? 0 : -1;  // 0=安全, -1=不安全
}
```

Done.md 实测经典 5 进程/3 资源案例，找安全序列 `<P1, P3, P4, P0, P2>`。时间复杂度 O(n²·m)。

### 第 15 页 — 哲学家就餐：从死锁复现到三层防护体系图（F15）

**功能示例**：实现哲学家就餐问题（死锁复现 + 三种防护）

**代码来源**：`user/dining.c`；`user/dining_safe1.c`；`user/dining_safe2.c`；`user/dining_auto.c`

**工作脉络**：把所有概念串联起来的最佳案例——**哲学家就餐问题**。从死锁复现，到三种防护手段，形成完整闭环。

**讲解词（约 55 秒）**：

> 最后用**哲学家就餐问题**把段 3 串联起来——从问题复现，到三种防护手段，形成完整的逻辑闭环。**这也是我们整个课程设计的"终局"：从进程管理→IPC→调度器→同步原语→死锁处理，这 15 张图的所有知识点都在这个问题上汇聚。**
>
> **图中央是原始死锁**：5 个哲学家同时拿起左叉，然后等待右叉——5 个进程、5 把叉子资源，形成循环等待。**死锁四要素全部满足**：互斥（叉子不可共享）、持有并等待（持左等右）、不抢占（叉子不可抢）、循环等待（P0等P1叉，P1等P2，...，P4等P0）。
>
> **左侧防护 A（打破 Hold & Wait）**：`dining_safe1.c` 引入 room 信号量（最多 4 人同时就座），进程**先抢座位再拿叉子**——持有资源前先申请了限制量，打破了"hold and wait"条件。
>
> **右侧防护 B（打破 Circular Wait）**：`dining_safe2.c` 强制所有哲学家**始终先拿编号小的叉子**，破坏了"循环等待"条件。
>
> **右侧防护 C（检测+恢复）**：`dining_auto.c` 不预防死锁，而是由内核的 deadlock_detect 每 30 ticks 扫描 Wait-For Graph（F13），发现环后 abort 最年轻的哲学家。
>
> **底部是三种方法对比表**：防护 A/B 需要事前知道死锁场景，防护 C 通用但有检测延迟，银行家算法在最前端阻止不安全分配（F14）。**我们的系统三层都有**，形成了完整的防护体系。

**逻辑图**（见 [F15](PPT_image/prompts/全部15张图提示词-学术黑白风.md#F15)）：

- 中心：5 个哲学家 × 5 把叉子，死锁状态（所有哲学家持左等右）
- 左：dining_safe1 时序（room semaphore 先于 fork）
- 右：dining_safe2 时序（编号小叉优先）+ dining_auto 内核检测恢复
- 底部：三列对比表（打破 hold-wait / 打破 circular-wait / 检测+恢复）

**文字说明**：

**死锁四必要条件**（dining.c 复现）：

| 条件 | dining.c 是否满足 |
|------|-----------------|
| 互斥 | ✓（叉子不可共享） |
| 持有并等待 | ✓（持左等右） |
| 不抢占 | ✓（叉子不可抢） |
| 循环等待 | ✓（P0等P1叉，P1等P2，...，P4等P0） |

**三层防护对比**：

| 防护 | 方法 | 打破条件 | 优点 | 缺点 |
|------|------|---------|------|------|
| A | room sem 限制≤4人 | hold-and-wait | 简单有效 | 需预知最大并发 |
| B | 编号排序拿小叉 | circular-wait | 无需额外资源 | 需预知资源顺序 |
| C | 内核死锁检测+abort | 无（事后恢复） | 最通用 | 有检测延迟 |
| Banker | request前安全检查 | 无（事先避免） | 零死锁 | 需预知最大需求 |

**测试结果**（Done.md §Phase B4）：

- dining.c：5 人全 SLEEPING，确认循环等待
- dining_safe1/2：正常完成，无死锁
- dining_auto：cycle=3，victim=pid5，4 人唤醒完成
