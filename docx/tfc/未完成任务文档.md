## 3. 当前已完成功能说明

### 3.1 PCB 与等待队列

当前 `struct proc` 已同时支持：
- 基础进程信息（PID、父子关系、页表、文件表）
- wait bucket 等待链表字段（`wnext`、`wprev`、`wbucket`）
- 调度扩展字段（`ctime`、`queue_level`、`timeslice_used`、`last_sched`、`priority`）

其中等待链表字段仅用于 `sleep()/wakeup()` 哈希等待桶，不再被 MLFQ 运行队列复用。

### 3.2 当前调度器状态

当前内核已经支持三种调度算法，且通过 `kernel/param.h` 中的 `SCHED_ALGORITHM` 宏在**编译期切换**：
- `SCHED_RR`
- `SCHED_FCFS`
- `SCHED_MLFQ`

当前实现特点：
- `RR`：按时间片轮转
- `FCFS`：按 `ctime` 选择最早创建的 RUNNABLE 进程
- `MLFQ`：通过 `queue_level` 与 `timeslice_used` 实现多级反馈调度

### 3.3 已有测试程序与结论

目前已有测试程序：
- `user/fcfstest.c`：支持验证 FCFS 的基本先来先服务顺序
- `user/mlfqtest.c`：支持验证 MLFQ 的短作业优先趋势
- `user/throughput.c`：目前只适合作为轻量性能观察
- `user/csw.c`：支持验证 RR/MLFQ 下的上下文切换路径

根据现有日志：
- `FCFS` 的基本顺序结论已成立
- `MLFQ` 的短作业优先趋势已成立
- `RR` 的轮转公平性已得到基本支持
- `throughput` 仍不足以支持高精度吞吐量排名

## 4. 仍未完成或仅部分完成的功能

### 4.1 MLFQ 可观测性增强

| 项目 | 说明 |
|------|------|
| **优先级** | 中 |
| **当前状态** | 基础功能已完成，但缺少直接观测证据 |
| **需求描述** | 为 MLFQ 增加可调试的队列迁移日志 |
| **原因** | 当前只能从现象推断 LONG 作业被降级，缺少直接日志证明 |

**建议增强：**
1. 在 MLFQ 模式下临时打印 `PID`、`queue_level`、`timeslice_used`
2. 记录进程被降级/提升的时刻
3. 形成更强的实验报告证据链

### 4.2 运行时时间片调节

| 项目 | 说明 |
|------|------|
| **优先级** | 低 |
| **当前状态** | 已支持编译期配置，未支持运行时调整 |
| **需求描述** | 时间片大小可在系统运行中动态调整 |
| **当前实现** | `TICKSLICE` 已在 `kernel/param.h` 中配置 |

**可选增强：**
1. 添加 `sys_settimeslice()` 系统调用
2. 提供运行时调整接口
3. 扩展实验对比不同时间片大小的影响

### 4.3 信号量机制

| 项目 | 说明 |
|------|------|
| **优先级** | 高 |
| **当前状态** | 未实现 |
| **需求描述** | 实现 POSIX 风格的信号量 |
| **实现思路** | 参照 `sleep()/wakeup()` 机制实现 P/V 操作 |

**建议数据结构**：
```c
struct semaphore {
  struct spinlock lock;
  int value;
  struct proc *owner;
};
```

**需实现：**
- `sem_init()`
- `sem_wait()`
- `sem_post()`
- 对应系统调用接口

### 4.4 互斥锁机制

| 项目 | 说明 |
|------|------|
| **优先级** | 高 |
| **当前状态** | 未实现 |
| **需求描述** | 实现进程间互斥锁 |
| **实现思路** | 可基于信号量封装，或单独实现 |

**建议数据结构**：
```c
struct mutex {
  struct spinlock lock;
  int held;
  struct proc *owner;
};
```

**需实现：**
- `mutex_init()`
- `mutex_lock()`
- `mutex_unlock()`
- 对应系统调用接口

### 4.5 性能评测增强

| 项目 | 说明 |
|------|------|
| **优先级** | 中 |
| **当前状态** | 已有轻量性能观察，缺少高精度对比 |
| **需求描述** | 形成更可靠的性能评测数据 |
| **当前限制** | `uptime()` 只有 tick 级分辨率，`throughput` 总时长仍偏短 |

**建议增强：**
1. 提高 `throughput.c` 中每个进程的工作量
2. 让 RR / FCFS / MLFQ 下总时长都达到几十 ticks 以上
3. 分别记录响应时间、公平性、总完成时间，而不只看单一吞吐量值

### 4.6 父子进程关系树与 `waitpid`

| 项目 | 说明 |
|------|------|
| **优先级** | 中 |
| **当前状态** | 部分实现（仅支持父进程指针与 `wait`） |
| **需求描述** | 完善进程树结构，提供按 PID 等待接口 |
| **当前限制** | 只能遍历 `proc[]` 查找子进程，未实现 `waitpid` |

**建议改进：**
1. 在 `struct proc` 中增加子进程链表或辅助结构
2. 实现 `waitpid(int pid)` 系统调用
3. 提供更完整的进程树遍历/展示能力

## 8. 当前结论摘要

1. `FCFS`、`RR`、`MLFQ` 三种调度器都已落地。
2. 时间片大小已经从硬编码改为 `kernel/param.h` 中的编译期配置。
3. `FCFS` 的先来先服务顺序、`MLFQ` 的短作业优先趋势、`RR` 的轮转公平性都已有日志支持。
4. 当前真正仍未完成的重点，已经从“调度器本身”转移到：
   - 同步原语（信号量、互斥锁）
   - `waitpid` / 进程树增强
   - 更强的性能评测与 MLFQ 可观测性
