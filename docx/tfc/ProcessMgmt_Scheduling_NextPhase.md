## 2. 下一阶段扩展蓝图（按知识板块组织）

### Phase A — 调度算法补齐（SJF / 优先级 / 时间片公平）

> **目标**：补齐 FCFS / RR / MLFQ 之外的两大经典算法，**丰富调度算法谱系**。

#### A1. SJF（Shortest Job First，非抢占式）

- **理论定位**：与 FCFS 对比，证明平均等待时间可显著降低
- **实现要点**：
  - 在 `struct proc` 新增 `uint64 est_burst`（估算的 CPU 突发时间）
  - 调度时选 `est_burst` 最小的 RUNNABLE 进程
  - 用户态提供 `setburst(pid, est)` 系统调用
  - 测试程序 `user/sjftest.c`：构造 4 个不同 burst 的进程，对比 FCFS / SJF 的总等待时间
- **预期结论**：SJF 显著优于 FCFS，但**可能长作业饥饿** → 自然引出 A2 的优先级继承问题

#### A2. 优先级调度（静态优先级 + 解决饥饿）

- **理论定位**：通用优先级调度的两个核心问题
  - 问题 1：低优先级饥饿
  - 问题 2：优先级反转（高优先级等低优先级持有锁）
- **实现要点**：
  - 复用 `struct proc` 已有的 `priority` 字段（0-10）
  - 优先级 + 时间片老化（aging）：等待时间越长优先级越高
  - **优先级继承（Priority Inheritance）**：当高优先级进程因低优先级持有锁而阻塞时，临时把低优先级进程的优先级提升到与高优先级相同
- **测试程序**：
  - `user/prioritytest.c`：验证 aging
  - `user/pi_test.c`：构造经典优先级反转场景，验证继承能缩短高优先级等待时间

#### A3. 时间片对 RR 公平性的影响

- **理论定位**：RR 的轮转周期 T = N × q，时间片太小 → 上下文开销大，太大 → 退化为 FCFS
- **实现要点**：
  - 复用 `settimeslice` / `gettimeslice` 系统调用
  - 测试程序 `user/rrqtest.c`：在 q=1, q=2, q=4 三种情况下，测 `csw` 总时间，画出 q 与吞吐的折线
- **教学价值**：把"运行时配置"和"调度效果"显式联系起来

### Phase B — 死锁专题（核心扩展点）

> **目标**：把"死锁"这一 OS 课程**最核心的同步问题**讲清楚、做完、做能跑。

#### B1. 死锁的 4 个必要条件 — 复现实验

- **理论**：互斥条件 / 占有并等待 / 不可剥夺 / 循环等待
- **实现**：
  - 用现有信号量构造"哲学家就餐"问题（5 个进程，5 把叉子）
  - 写 `user/dining.c`：演示若 5 个哲学家同时拿起左侧叉子 → 死锁
- **结论**：直接观察到 `ps` 中 5 个进程长期处于 SLEEPING，调度器无进展

#### B2. 死锁预防（破坏 4 条件之一）

- **方案 A — 破坏"占有并等待"**：进程一次性申请所有资源
  - 写 `user/dining_safe1.c`：使用一个 `room` 信号量，最多允许 4 个哲学家进入
- **方案 B — 破坏"循环等待"**：给资源编号，强制按编号顺序申请
  - 写 `user/dining_safe2.c`：奇数号哲学家先拿左再右，偶数号先右后左
- **测试**：在两个 safe 版本中，连续运行 1000 次，统计死锁次数 → 0

#### B3. 死锁避免 — 银行家算法（核心）

- **理论**：安全状态 / 安全序列 / 工作向量 / Need 矩阵
- **数据结构**：
  ```c
  // kernel/banker.h
  #define NRES 8           // 资源种类数
  struct banker_state {
    int available[NRES];
    int max[NRES];         // 进程最大需求（每个进程一份）
    int allocation[NRES];  // 进程已分配
    int need[NRES];        // max - allocation
  };
  ```
- **系统调用**：
  - `sys_banker_init(nres, avail[])` — 初始化资源向量
  - `sys_banker_request(pid, req[])` — 申请资源（含安全检查）
  - `sys_banker_release(pid, rel[])` — 释放资源
- **核心算法**（用户态可重写验证）：`is_safe_state()` 实现安全性算法
- **测试程序**：
  - `user/bankertest.c`：经典 5 进程 3 资源示例，验证能找到安全序列
  - `user/banker_unsafe.c`：构造一个让算法拒绝的不安全请求
- **教学价值**：这是 OS 课程**最具代表性的算法**，必须实现

#### B4. 死锁检测 + 恢复（最小实现）

- **理论**：用资源分配图 / 等待图检测环
- **实现**：
  - 内核周期性（每 N tick）扫描 `wait_for_lock` 关系，建等待图
  - 用 DFS 检测环，发现死锁后**剥夺（abort）最年轻进程**并释放其资源
- **测试**：让哲学家就餐死锁自动恢复，输出 `[DEADLOCK] abort pid=X`

### Phase C — 高级同步原语（管程 / 条件变量）

> **目标**：与信号量并列的另一种同步范式，**抽象层级更高**。

#### C1. 管程（Monitor）的基本结构

- **理论**：Hoare 管程 vs Mesa 管程；条件变量 `wait` / `signal` vs `broadcast`
- **数据结构**：
  ```c
  // kernel/monitor.h
  #define NMON 8
  struct monitor {
    char name[16];
    struct spinlock lock;        // 管程互斥锁
    int owner;                   // 当前持有管程的进程 pid
    struct cv cond[NCOND];       // 条件变量
  };
  struct cv {
    int signaled;                // 是否有 signal 到来（Mesa 风格）
    struct spinlock lock;
    struct proc *waiters;
  };
  ```
- **系统调用**：
  - `sys_mon_create(name)` / `sys_mon_enter(mid)` / `sys_mon_exit(mid)`
  - `sys_cv_wait(mid, cvid)` / `sys_cv_signal(mid, cvid)` / `sys_cv_broadcast(mid, cvid)`

#### C2. 用管程重写生产者-消费者

- 把 `semtest3` 改为 `monitortest.c`，3 个生产者 / 3 个消费者
- 对比信号量版本和管程版本的代码复杂度
- **教学价值**：体现"管程把锁的细节封装起来，程序员只需关注同步条件"

### Phase D — 进程高级主题

#### D1. 进程优先级反转与优先级继承（再次强调）

- 与 A2 联动，但在 Phase D 中作为**独立主题**深化
- 写一个**经典 Mars Pathfinder Bug 复现实验**：
  - 高、中、低三优先级进程，低优先级持有锁
  - 中优先级不持锁但消耗 CPU
  - 观察：高优先级因等待低优先级而无法运行
  - 加入优先级继承后：高优先级直接继承低优先级的优先级，抢占中优先级
- **测试程序** `user/pathfinder.c`

#### D2. 进程通信的更多模型

- **消息队列（message queue）**：
  - 系统调用 `mq_send / mq_recv`，内核维护一个简单的固定大小环形队列
  - 测试 `user/mqtest.c`
- **管道增强**：
  - 复用 xv6 原生 `pipe`，写一个跨管道的生产者-消费者基准
  - 测试 `user/pipebench.c`

### Phase E — 多处理机调度（高级扩展）

> **目标**：利用现有 `NCPU=8` 配置，把调度器扩展到**多核场景**。

#### E1. Per-CPU 调度队列

- **现状**：所有 CPU 共用一个 `proc[]` 数组 + 单一 `scheduler()`
- **改造**：
  - 每个 CPU 维护自己的 `struct proc *runq[NRUN]` 局部就绪队列
  - 进程唤醒时通过当前 CPU 入队
  - 调度时只遍历当前 CPU 的局部队列
- **理论**：
  - 降低锁粒度（每 CPU 一把 `runq_lock`）
  - 解决"调度器成为瓶颈"问题

#### E2. 负载均衡（Load Balancing）

- **理论**：各 CPU 队列长度差异大时需要迁移
- **策略**：
  - **Pull 策略**：空闲 CPU 从繁忙 CPU 拉一个进程
  - 每 N tick 触发一次
- **测试**：
  - `user/mp_bal.c`：8 个 worker 进程 + 8 个 sleep 进程，观察各 CPU 的 `runq` 长度

#### E3. 多核下的同步原语验证

- **自旋锁压测** `user/spinbench.c`：8 个线程同时争抢一把自旋锁
- **信号量压测**：8 个 producer / 8 个 consumer 看吞吐
- **结论**：
  - 信号量在内核态实现，跨 CPU 也能工作
  - 自旋锁适合短临界区，长临界区会浪费 CPU

### Phase F — 实时调度（选修）

#### F1. Rate-Monotonic（RM）

- **理论**：周期越短优先级越高；可调度条件 ∑(Cᵢ/Tᵢ) ≤ n(2^(1/n) - 1)
- **实现**：
  - 系统调用 `sys_rt_register(period, cost)` — 注册一个实时任务
  - 内核为实时任务维护独立的优先级表
  - 调度时优先选实时任务

#### F2. Earliest Deadline First（EDF）

- **理论**：动态优先级 = 截止时间最近
- **实现**：用截止时间最小堆

#### F3. 测试 `user/rmtest.c` & `user/edftest.c`

- 构造 3 个周期性任务
- 验证在最坏情况下截止时间达成率

## 6. 不在本阶段范围内（明确排除）

- 文件系统 / 设备驱动 / 虚拟内存 → 属其他章节
