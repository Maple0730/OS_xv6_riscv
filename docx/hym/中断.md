## 一、概述

xv6-riscv 运行在 RISC-V 架构上，使用 **`stvec` 单入口 + `scause` 软件分发** 的模式处理所有中断与异常。这与 x86 的内存驻留 IDT（256 条目数组）不同——RISC-V 的中断向量机制更简洁：硬件只负责跳到 `stvec` 指向的入口地址，由软件读取 `scause` 寄存器来区分中断/异常类型并分发到对应处理函数。

### RISC-V vs x86 对比

| | x86 IDT | RISC-V stvec（xv6 用法） |
|---|---|---|
| 数据结构 | 内存中 256 条目的数组 | 单个 CSR 寄存器，存入口地址 |
| 硬件行为 | 硬件按中断号查表跳转 | 全部 trap 跳到同一地址 |
| 分发机制 | **硬件**查表分发 | **软件**读 `scause` 分发 |
| 课程对应 | 初始化 IDT | 初始化中断向量表 |

## 二、中断向量表的三层设置

### 2.1 第一层：M 态委托（`kernel/start.c`）

系统启动时，CPU 处于 M 态（Machine Mode）。`start()` 负责将中断和异常委托给 S 态（Supervisor Mode），并初始化定时器。

```c
// start.c:30-42
// 委托所有异常（除 ecall from S-mode）到 Supervisor
w_medeleg(0xffff & ~(1L << 9));
// 委托所有中断到 Supervisor
w_mideleg(0xffff);
// 开启 S 态外部中断 + 时钟中断
w_sie(r_sie() | SIE_SEIE | SIE_STIE);
// 配置 PMP 允许 S 态访问全部物理内存
w_pmpaddr0(0x3fffffffffffffull);
w_pmpcfg0(0xf);
// 启动时钟中断
timerinit();
```

**关键点**：
- `medeleg`/`mideleg` 允许 S 态直接处理异常和中断，无需 M 态中转
- `bit 9`（ecall from S-mode）不委托，保留在 M 态用于 SBI（Supervisor Binary Interface，如 `sbi_shutdown()`）
- `timerinit()` 通过 SSTC 扩展（`stimecmp`）设置第一个时钟中断

```c
// start.c:53-64 — 时钟中断初始化
void timerinit() {
    w_menvcfg(r_menvcfg() | (1L << 63));  // 开启 sstc 扩展
    w_mcounteren(r_mcounteren() | 2);     // S 态可访问 time CSR
    w_stimecmp(r_time() + 1000000);        // 首次中断：~10ms 后（100Hz）
}
```

**文件**：`kernel/start.c`（15-64 行）

### 2.2 第二层：内核态 trap 向量（`kernel/trap.c` + `kernel/kernelvec.S`）

当 CPU 进入 S 态后，`main()` 调用 `trapinithart()` 设置内核态 trap 入口：

```c
// trap.c:26-30
void trapinithart(void) {
    w_stvec((uint64)kernelvec);  // 内核态全部 trap → kernelvec
}
```

`kernelvec`（汇编）的功能：
1. 在**内核栈**上保存 caller-saved 寄存器（ra、gp、a0-a7、t0-t6 等）
2. 调用 C 语言处理函数 `kerneltrap()`
3. 恢复寄存器
4. `sret` 返回被中断的代码

```asm
// kernelvec.S:12-64 — 内核 trap 入口
kernelvec:
    addi sp, sp, -256         // 在栈上开 256 字节空间
    sd ra, 0(sp)              // 保存寄存器
    // ... 保存 gp, t0-t6, a0-a7 ...
    call kerneltrap            // → trap.c:149
    ld ra, 0(sp)              // 恢复寄存器
    // ... 恢复各寄存器 ...
    addi sp, sp, 256          // 回收栈空间
    sret                      // 返回
```

`kerneltrap()`（trap.c:149）处理内核态下的中断：
- `devintr()` 检测是否为设备中断（时钟/外设）
- 若是时钟中断 → `yield()` 让出 CPU
- 否则 → `panic("kerneltrap")`

```c
// trap.c:149-170
void kerneltrap() {
    int which_dev = 0;
    uint64 sepc = r_sepc();
    uint64 sstatus = r_sstatus();
    uint64 scause = r_scause();

    if ((sstatus & SSTATUS_SPP) == 0)
        panic("kerneltrap: not from supervisor mode");

    if (kerneltrap_is_broken(scause))
        panic("kerneltrap: kernel page fault");

    which_dev = devintr();   // 检查是不是设备中断

    if (which_dev == 0)
        panic("kerneltrap: unexpected scause %p", scause);

    if (which_dev == 2)      // 时钟中断
        yield();

    w_sepc(sepc);            // 恢复 sepc
    w_sstatus(sstatus);      // 恢复 sstatus
}
```

**文件**：`kernel/trap.c`（26-30 行，149-170 行），`kernel/kernelvec.S`（1-64 行）

### 2.3 第三层：用户态 trap 向量（`kernel/trampoline.S`）

返回用户态前，`usertrap()` 将 `stvec` 指向 `uservec`（位于 trampoline 页）：

```c
// trap.c:127
w_stvec(trampoline_uservec);  // 用户态 trap → uservec
```

**trampoline 机制**（`trampoline.S`）：
- trampoline 页面被同时映射到**用户页表和内核页表的同一虚拟地址**（`TRAMPOLINE`）
- 切换到内核页表时，trampoline 代码仍然可执行（因为地址不变）
- 解决了"用户态 trap 发生后不知道内核页表地址"的问题

**用户态 trap 完整路径**：

```
用户程序执行
  ↓ (中断/异常/ecall 发生)
CPU 硬件:
  保存 pc → sepc
  保存 mode → sstatus.SPP
  有中断原因 → scause
  切换到 S 态
  跳到 stvec → uservec (trampoline.S)
  ↓
uservec (汇编):
  1. 保存全部 32 个用户寄存器到 TRAPFRAME
  2. 从 TRAPFRAME 加载: 内核栈指针、hartid、usertrap 地址、内核 satp
  3. sfence.vma + csrw satp 切换到内核页表
  4. jalr 跳转到 usertrap() (C 代码)
  ↓
usertrap() (trap.c:38):
  w_stvec(kernelvec)              // 在内核态期间，用 kernelvec
  读 scause 判断类型:
    scause == 8  → syscall()      // ecall 系统调用
    devintr()    → 时钟/设备中断
    其他         → vmfault() 等异常处理
  ↓
  usertrapret() (trap.c:93):
    设置 sepc 指向返回地址
    设置 sstatus 切回用户态
    w_stvec(uservec)              // 切回用户态前恢复
  ↓
userret (trampoline.S:101):
  1. 切回用户页表 (csrw satp, a0)
  2. 从 TRAPFRAME 恢复全部用户寄存器
  3. sret → 回到用户程序
```

**文件**：`kernel/trampoline.S`（1-149 行），`kernel/trap.c`（38-93 行）

## 三、软件分发机制：`devintr()` 与 `scause`

`devintr()`（`kernel/trap.c:183`）是中断类型识别的核心，通过读取 `scause` 寄存器来区分中断来源：

```c
// trap.c:183-220
int devintr() {
    uint64 scause = r_scause();

    // Supervisor 外部中断（UART、VIRTIO、PLIC）
    if ((scause & 0x8000000000000000L) &&
        (scause & 0xff) == 9) {
        // PLIC 扫描确认具体设备
        int irq = plic_claim();
        if (irq == UART0_IRQ) {
            uartintr();           // UART 中断 → 键盘输入
        } else if (irq == VIRTIO0_IRQ) {
            virtio_disk_intr();   // 磁盘中断
        }
        // ... 其他 PLIC 设备 ...
        plic_complete(irq);       // 告知 PLIC 处理完成
        return 1;
    }

    // 时钟中断（mtimecmp 到期）
    else if ((scause & 0x8000000000000000L) &&
             (scause & 0xff) == 5) {
        // 编程下一次时钟中断
        w_stimecmp(r_time() + 1000000);  // +10ms
        return 2;
    }

    return 0;  // 不是设备中断
}
```

### scause 关键值

| scause 值 | 类型 | 含义 |
|-----------|------|------|
| `0x8000000000000009` | 中断 | Supervisor 外部中断 |
| `0x8000000000000005` | 中断 | Supervisor 时钟中断 |
| `8` | 异常 | ecall from U-mode（系统调用） |
| `13` | 异常 | Load page fault（缺页异常） |
| `15` | 异常 | Store page fault（缺页异常） |

## 四、三类中断/异常的完整处理路径

### 4.1 时钟中断 → 进程调度

```
硬件定时器到期
  → stvec → uservec/kernelvec
  → usertrap()/kerneltrap()
  → devintr() 识别 scause=5
  → w_stimecmp(r_time() + 1000000)   // 设置下次中断
  → yield()                           // 让出 CPU
    → sched() → swtch()               // 上下文切换
```

**频率**：约 100Hz（每 ~10ms 触发一次），`timerinit()` 中 `stimecmp = time + 1000000` ticks。

### 4.2 键盘中断 → 控制台输入

```
用户按键
  → UART 硬件收到字符
  → PLIC 路由中断到 hart
  → stvec → uservec/kernelvec
  → devintr() → plic_claim() → irq == UART0_IRQ
  → uartintr()                       // kernel/uart.c
    → uartgetc() 读取字符
    → consoleintr(c)                 // kernel/console.c
      → 特殊键处理（退格、Ctrl-U、Ctrl-D、Ctrl-P）
      → cons.buf[] 累积
      → 遇 '\n' 或 Ctrl-D → wakeup(&cons.r)
        → 唤醒等待 read() 的 shell 进程
```

### 4.3 系统调用（ecall）

```
用户程序: printf("hello")
  → write(1, "hello", 5)
  → user/usys.S 生成: li a7, SYS_write; ecall; ret

ecall 陷入内核:
  → stvec → uservec (trampoline.S)
  → usertrap()
  → scause == 8 → syscall()          // kernel/syscall.c
    → 读 a7 (SYS_write=16)
    → syscalls[16] → sys_write()     // kernel/sysfile.c
      → filewrite()                  // kernel/file.c
      → consolewrite()               // kernel/console.c （带 writelock）
    → trapframe->a0 = 返回值
  → usertrapret()
  → sret → 回到用户态

返回值通过 p->trapframe->a0 传回用户态。
```

### 4.4 缺页异常 → 懒分配

```
用户程序: sbrklazy(4096) 申请内存（只扩大 p->sz，不分配物理页）
  → 首次访问该页
  → MMU 查页表 → PTE 不存在 → page fault
  → stvec → uservec
  → usertrap() → scause=13/15
  → vmfault()                        // kernel/vm.c
    → kalloc() 分配物理页
    → memset 清零
    → mappages() 建立映射
  → 返回用户态，重新执行触发缺页的指令
```

**文件**：`kernel/vm.c`（`vmfault()`），`kernel/trap.c`（71-73 行）

## 五、PLIC（平台级中断控制器）

RISC-V 使用 PLIC 管理外部设备中断：

```
UART ──┐
VIRTIO ─┼── PLIC ── 路由到 hart 0
其他设备─┘
```

关键操作：
- **初始化**：`plicinit()` + `plicinithart()`（`kernel/plic.c`）
- **中断到达**：`plic_claim()` 读取中断源（返回 IRQ 号）
- **中断完成**：`plic_complete(irq)` 告知 PLIC 可以接收下一个中断
