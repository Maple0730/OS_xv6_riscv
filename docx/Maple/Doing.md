## 内存管理

### 一、实现内核堆内存分配（`kmalloc/kfree`）

#### 1. 当前问题

当前内核只有页级分配器：

- `kalloc()`：按页分配 `4KB` 物理内存
- `kfree()`：按页回收 `4KB` 物理内存

这意味着：

- 页表页、用户页、内核栈这类“整页对象”可以直接分配
- 但几十字节、几百字节的小型内核对象没有独立的动态分配接口
- 如果强行用 `kalloc()` 处理小对象，会造成明显的页内浪费

因此，需要在 `kalloc/kfree` 之上补一层**内核堆分配器**，实现 `kmalloc/kfree`。

#### 2. 总体目标

目标不是替换当前页分配器，而是增加一层“小块动态分配”能力：

- `kalloc/kfree` 继续负责页级物理内存管理
- `kmalloc/kmfree` 负责内核中非页大小对象的分配与回收

最终形成两层结构：

- 页级分配器：`kalloc/kfree`
- 堆级分配器：`kmalloc/kmfree`

#### 3. 技术路线选择

计划采用**简单 slab / size-class 分配器**，而不是通用自由链表分配器。

选择原因：

- 更符合 xv6 当前内核场景
- 可直接复用现有 `kalloc()` 提供的整页内存
- 实现复杂度低，便于验证和讲解
- 比照搬用户态 `umalloc` 更适合内核

基本思想：

- 按若干固定大小档位管理小对象
- 每个档位从 `kalloc()` 申请一页
- 再将这一页切分成多个等大小对象
- `kmalloc()` 从对应档位的空闲链表中取对象
- `kmfree()` 将对象归还到对应档位

#### 4. 初步分配档位

首版计划使用以下 size class：

- `32`
- `64`
- `128`
- `256`
- `512`
- `1024`
- `2048`

分配策略：

- 请求大小 `n` 落到第一个 `size >= n` 的档位
- 例如：
  - `kmalloc(20)` 走 `32B`
  - `kmalloc(100)` 走 `128B`
  - `kmalloc(700)` 走 `1024B`

首版边界策略：

- `kmalloc(0)` 返回 `0`
- 大于 `2048` 的请求直接返回 `0`
- 首版不处理“大对象自动退回整页分配”，以简化实现和释放逻辑

#### 5. 数据结构设计

每个 size class 维护：

- 一把 `spinlock`
- 一个空闲对象链表
- 对应档位对象大小
- 已申请页数、对象总数、空闲对象数等统计信息

每个被切分的页页头维护一个 `slab_page` 元数据，至少包含：

- 所属 size class 编号
- 当前页对象总数
- 当前页空闲对象数
- 当前页空闲链表头
- 下一页指针

设计要点：

- 空闲对象本身可复用前几个字节作为 `next` 指针
- 页头存放页级元数据
- 页头后的剩余空间按固定对象大小切分
- 对象起始地址要按指针大小对齐

#### 6. 分配流程

`kmalloc(n)` 的计划流程：

1. 判断 `n` 是否合法
2. 根据大小找到目标 size class
3. 若该档位已有空闲对象，则直接取出返回
4. 若该档位没有空闲对象，则：
   - 调用 `kalloc()` 申请一页
   - 初始化该页页头元数据
   - 将该页切分为多个固定大小对象
   - 将对象挂入该档位空闲链表
5. 从新补充的空闲链表中取出一个对象返回

初始化策略：

- 新切分出的对象默认清零，保证行为稳定
- 新页也可按调试需要填充特定模式，便于发现未初始化使用问题

#### 7. 释放流程

`kmfree(p)` 的计划流程：

1. 若 `p == 0`，直接返回
2. 通过页对齐方式找到 `p` 所属页
3. 读取该页页头元数据，判断所属 size class
4. 检查指针是否位于对象区，且满足对象大小对齐要求
5. 将该对象插回该页空闲链表
6. 更新该页和该 size class 的空闲统计

首版策略：

- 非法指针直接 `panic("kmfree")`
- 首版不把“已经全部空闲的 slab 页”归还给 `kfree()`
- 优先保证分配/释放路径正确，再考虑整页回收优化

#### 8. 代码组织与接入点

计划新增接口：

- `kheapinit()`
- `kmalloc(uint n)`
- `kmfree(void *p)`

计划新增或修改位置：

- 新增 `kernel/kmalloc.c`：实现 `kmalloc/kmfree/kheapinit`
- 修改 `kernel/defs.h`：增加函数声明
- 修改 `kernel/main.c`：在 `kinit()` 之后初始化 `kheapinit()`
- 修改 `Makefile`：将 `kernel/kmalloc.c` 编入构建

首轮接入策略：

- 先只实现并接入 `kmalloc/kfree` 机制本身
- 先不大规模替换现有 `kalloc()` 调用点
- 等机制稳定后，再挑选真正的小对象使用场景逐步迁移

这样做的原因：

- 可以降低回归风险
- 便于单独验证 `kmalloc/kfree` 的正确性
- 避免把“实现内核堆”和“改造现有调用点”混成一个问题

#### 9. 验证思路

至少验证以下几类行为：

- `kmalloc(1)`、`kmalloc(32)`、`kmalloc(33)`、`kmalloc(2000)` 可正常返回
- `kmalloc(2049)` 返回失败
- 多次申请和释放后，对象地址能够复用
- 不同档位对象混合申请/释放时互不干扰
- 单页对象用尽后可自动申请新页继续切分
- `kmfree(0)` 安全返回
- `kmfree()` 对非法指针能报错或 panic

#### 10. 当前取舍

为了控制复杂度，首版明确不做以下内容：

- 通用 malloc 风格自由链表
- 大对象自动回退整页分配
- 空 slab 页自动归还页分配器
- 复杂碎片整理
- 大规模替换现有 `kalloc()` 调用点

这一版的重点是：

- 把 `kmalloc/kfree` 作为独立内核堆机制先做对
- 让它能够稳定地服务小块内核对象分配
- 为后续 `Copy-on-Write`、内存状态观测等功能提供更好的底层支持

#### 11. 测试使用教程

当前版本已经在内核启动早期接入了 `kmalloctest()` 自测函数。

接入位置：

- `kernel/main.c`
- 启动顺序中位于 `kinit()` 和 `kvminit()` 之间

也就是说：

- 每次启动内核时
- `kheapinit()` 完成后
- 会自动执行一次 `kmalloc/kfree` 自测

推荐使用方法：

1. 重新编译内核：

```bash
make
```

2. 启动 xv6：

```bash
make qemu
```

3. 观察启动日志中的 `kmalloctest` 输出

如果只想做一次短时间验证，也可以使用：

```bash
timeout 10s make qemu
```

当前自测覆盖的项目包括：

- 边界值测试：
  - `kmalloc(0)` 应返回 `0`
  - `kmalloc(2049)` 应返回 `0`
- 基本分配测试：
  - `kmalloc(1)`
  - `kmalloc(32)`
  - `kmalloc(33)`
  - `kmalloc(2000)`
- 可写性测试：
  - 对已分配对象写入并读回，确认对象可正常使用
- 复用测试：
  - 释放对象后再次申请，确认地址可复用
- 跨页扩展测试：
  - 连续申请 `140` 个 `32B` 对象，强制触发一个档位跨越单页容量并扩容
- 安全测试：
  - `kmfree(0)` 安全返回

#### 12. 当前测试结果

当前一次典型启动日志如下：

```text
xv6 kernel is booting

kmalloctest: begin
kmalloctest: boundary checks ok
kmalloctest: basic allocations ok
kmalloctest: read/write checks ok
kmalloctest: reuse check ok
kmalloctest: cross-page growth ok (140 objects)
kmalloc stats:
  class  size  pages  total  free  used
  0 32 2 254 254 0
  1 64 1 63 63 0
  2 128 0 0 0 0
  3 256 0 0 0 0
  4 512 0 0 0 0
  5 1024 0 0 0 0
  6 2048 1 1 1 0
  alloc_ok=145 alloc_fail=1 free_ok=145
  total_pages=4 total_objs=318 free_objs=318 used_objs=0
kmalloctest: ok
```

说明：

- `boundary checks ok`
  - 表示越界输入和空输入路径符合预期
- `basic allocations ok`
  - 表示多个不同档位都能成功返回可用对象
- `read/write checks ok`
  - 表示返回对象不仅非空，而且可安全读写
- `reuse check ok`
  - 表示释放后对象确实回到了对应空闲链表，可再次复用
- `cross-page growth ok`
  - 表示 `32B` 档位对象用尽后，可以自动向页分配器申请新页并继续切分
- `kmalloctest: ok`
  - 表示本轮自测整体通过，没有触发 panic

#### 13. 统计结果分析

从当前统计结果可以读出以下信息：

1. `32B` 档位：
   - 使用了 `2` 页
   - 共切分出 `254` 个对象
   - 当前全部已释放，因此 `free=254`、`used=0`
   - 这与“连续申请 `140` 个 `32B` 对象”这一测试是匹配的，说明单页容量不足后确实触发了扩页

2. `64B` 档位：
   - 使用了 `1` 页
   - 共切分出 `63` 个对象
   - 当前已全部释放
   - 这对应 `kmalloc(33)` 被提升到 `64B` 档位

3. `2048B` 档位：
   - 使用了 `1` 页
   - 共切分出 `1` 个对象
   - 当前已全部释放
   - 这对应 `kmalloc(2000)` 走 `2048B` 档位

4. `alloc_ok=145`
   - 表示本轮共有 `145` 次成功分配
   - 其中包括基本分配、复用测试中的重新申请，以及跨页扩展测试中的批量申请

5. `alloc_fail=1`
   - 表示本轮有 `1` 次失败分配
   - 这正是预期中的 `kmalloc(2049)` 越界请求

6. `free_ok=145`
   - 表示所有成功申请的对象最终都完成了回收
   - 与 `alloc_ok=145` 对应

7. `used_objs=0`
   - 表示测试结束后没有遗留未释放对象
   - 从当前自测路径看，分配与释放数量匹配

#### 14. 当前结论

基于当前自测结果，可以得到以下阶段性结论：

- `kmalloc/kfree` 首版机制已经能够正常工作
- 固定档位分配策略已经打通
- 对象可写、可释放、可复用
- 一个档位耗尽后能够自动向 `kalloc()` 申请新页继续扩展
- 当前自测路径结束后，对象层面没有残留未释放项

但也需要保持表述准确：

- 现在证明的是“当前自测覆盖路径工作正常”
- 还不能据此推出“整个系统所有未来使用场景都绝对无 bug”
- 后续仍需要把 `kmalloc/kfree` 逐步接入真实内核对象场景，再继续验证
