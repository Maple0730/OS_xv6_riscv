# 01_cover
这一页先交代汇报边界：我负责的是系统启动、内存系统和文件系统三个模块。要强调本次汇报不是从零重写一个操作系统，而是在 xv6-riscv 教学内核基础上完成理解、扩展和验证。最后一句自然过渡到下一页，说明接下来先给出个人负责范围和汇报组织方式。

# 02_overview
这一页把结构说清楚。先指出三个负责模块，再说明每个模块都按照“整体框架、基础功能、扩展实现、测试验证”四层来讲，这样老师容易对应课程要求和个人工作。讲完后过渡到系统启动，说明先从最早发生的链路开始。

# 03_boot_overview
这一页重点讲启动主链路。先顺着图从 `_entry` 说到 `start()`、`main()`、`userinit()`、`init` 和 `sh`，再解释 QEMU 如何把内核加载到 0x80000000，以及 `entry.S` 和 `start()` 分别承担什么责任。结尾点出：有了这条主链路，才谈得上后面的内存和文件系统初始化。

# 04_boot_basics
这一页对应课程要求里的基础启动能力。讲法建议是四点并列：M 态到 S 态切换、每个 hart 的启动栈准备、trap vector 初始化，以及最终进入 `main()` 和 shell。语速保持平稳，不需要展开每个寄存器细节，只要让老师知道这些指标都有明确实现位置即可。最后过渡到启动阶段上我补充的三个扩展点。

# 05_boot_extensions
这一页讲启动阶段的三个扩展。先说保存 DTB 指针，这是后面运行时物理内存探测的输入；再说把 `kmalloctest` 接进启动日志，便于在内核早期发现堆分配问题；最后说 shell 默认进入 `/desktop`，把根目录留给系统入口和挂载点。可以顺手指一下右侧截图，让老师看到这些改动是可见的、不是停留在代码里。

# 06_memory_overview
这里开始内存系统。先把整张图拆成三层：物理页框分配、虚拟内存管理、内核堆分配器。说明这三层不是并列孤岛，而是自底向上依赖：页框是底座，页表负责地址空间，`kmalloc` 负责对象级精细分配。结尾提示下一页先看 xv6 原有的基础能力，再看我补上的扩展。

# 07_memory_basics
这一页讲内存系统已有的基础能力。建议按四块说明：页级分配、Sv39 页表与映射、用户空间布局，以及 lazy `sbrk` 的按需分页。这里的重点不是夸大“我从零实现了什么”，而是把平台原有能力梳理清楚，为后面的扩展定位坐标。最后过渡到：我的核心工作在运行时探测和 `kmalloc`。

# 08_memory_extensions
这一页是内存部分的重点。先讲运行时物理内存探测，强调不再写死物理内存上界，而是通过 DTB `/memory/reg` 读取 RAM 范围，所以在不同 QEMU 内存规模下都能自适应。然后讲 `kmalloc/kmfree` 两级架构，说明 7 个 slab 档位、小对象路径、大对象整页回退，以及 `pipe`、`file` 已经接入真实主路径。收尾用一句话概括：这让系统同时具备了规模自适应和对象级精细分配能力。

# 09_memory_tests
这一页专门把“实现了”和“跑通了”区分开。先看左侧启动日志里的 `kmalloctest: ok`，再说明它覆盖了边界检查、读写检查、复用检查和跨页增长检查。接着指出 `pipe` 和 `file` 对象已经改走 `kmalloc`，说明这不是单独的样例函数，而是进入了真实运行路径。最后过渡到文件系统部分。

# 10_fs_overview
文件系统部分先讲整体结构。建议从最上层的 `sysfile.c` 一直讲到底层的 virtio disk，把 `file.c`、`fs.c`、`log.c`、`bio.c` 串成一条链。这里要让老师建立一个认知：后面的 `lseek`、双盘和隐藏显示，都是在这个主干上的不同层次做改动。

# 11_fs_basics
这一页讲基础文件系统功能。先从进程视角解释 `fd -> struct file -> inode -> disk block` 这条路径，再说明基本文件/目录操作和路径解析都是可用的。最后补充 `lseek`，指出它支持 `SEEK_SET`、`SEEK_CUR`、`SEEK_END`，因此文件系统已经具备随机访问能力。结尾自然过渡到扩展一，也就是日志和双独立磁盘。

# 12_fs_log_dualdisk
这一页有两个核心点。第一，`log.c` 保证多块更新的事务一致性，可以用 `begin_op/end_op`、`log_write/commit/recover_from_log` 这条主线讲清楚。第二，双独立磁盘不是简单多挂一个镜像，而是把 superblock、inode、日志和路径解析都按设备号解释，同时通过 `/disk1` 提供命名空间入口。最后强调一句：这部分创新点在于“按设备分流”，而不是“多一个 .img 文件”。

# 13_fs_hidden_color
这一页讲用户可见的目录浏览增强。先讲隐藏规则：`.`、`..`、点文件和根目录预装系统文件默认隐藏，`ls -a` 再展开。再讲颜色规则：目录蓝色、设备文件黄色、隐藏项灰色。最后一定要点明实现位置在用户态 `ls`，所以没有改磁盘格式，也没有给 inode 增加新属性，这一点能体现设计上的克制。

# 14_fs_tests
这一页把文件系统扩展的验证收束起来。先看左边的 `lseektest` 截图，说明三种 whence 和文件大小获取都已通过。再看右边隐藏与颜色显示截图，说明展示策略确实生效。双盘测试虽然不在截图里，但要口头补一句：在 `/disk1` 路径下的 `open/create/write/read` 和日志提交都已经跑通，因此双盘不是停留在设备识别层。

# 15_summary
最后一页不要简单说“谢谢”。建议按三个模块各用一句话回顾：启动链路稳定、内存系统具备运行时探测和两级分配、文件系统扩展了 `lseek`、日志、双盘和显示增强。结尾一句落在“这些工作都已经通过真实运行验证”，把整场答辩收束到“可复现、可证明”的结论上。
