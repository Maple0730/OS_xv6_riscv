# 韩硕个人 PPT 配图方案

> 适用对象：`韩硕个人 PPT`
>
> 对应提纲：`docx/Maple/韩硕个人PPT提纲.md`
>
> 目标：在保证技术内容准确的前提下，尽量复用现有资源，减少重复做图工作。

---

## 一、总原则

这套 PPT 的图不建议全部依赖 AI 生成。

更稳妥的做法是：

- 截图类页面：直接复用现有截图
- 流程图/结构图页面：优先使用 `Mermaid` 或手工框图
- 封面页或装饰性概念图：可以使用 `imagegen`

原因很简单：

- 技术结构图要求箭头关系准确、文字准确
- AI 位图更适合做“风格图”和“概念背景图”
- 现有仓库里已经有一批可以直接使用的截图和图表，不需要重新折腾

## 三、各页配图方案

下面按你提纲里的页次逐页安排。

---

### 第 1 页：封面

**建议图类型**：

- 抽象技术风封面图

**推荐方案**：

- 可选用 `imagegen` 生成一张科技感背景图
- 也可以不用图，直接采用深蓝标题条 + 白底版式

**适合放的视觉元素**：

- RISC-V / 内核 / 页表 / 磁盘块 / 路径结构
- 只做背景暗示，不要放大量可读文字

**结论**：

- 这页如果要用 AI，`imagegen` 最合适
- 但不是必须项

---

### 第 2 页：个人负责内容总览

**建议图类型**：

- 不需要大图

**推荐方案**：

- 直接做三栏结构：
  - 系统启动
  - 内存系统
  - 文件系统

**结论**：

- 这一页不建议用 `imagegen`
- 直接用 PPT 自己画模块框最快

---

### 第 3 页：系统启动整体框架

**建议图类型**：

- 启动流程图

**推荐方案**：

- 用 `Mermaid` 或 PPT 框图手工绘制

**建议结构**：

```text
QEMU
  -> _entry
  -> start()
  -> main()
  -> userinit()
  -> init
  -> sh
```

**图中建议强调**：

- `0x80000000`
- `entry.S`
- `M -> S`
- 最终进入 shell

**结论**：

- 这页不要用 AI 位图
- 结构图必须自己可控

---

### 第 4 页：系统启动基础功能

**建议图类型**：

- 无主图，使用表格或对照框

**推荐方案**：

- 左侧列“功能要求”
- 右侧列“当前实现”

可列：

- 特权级切换
- 栈初始化
- trap 向量
- 进入 `main()`
- 创建第一个用户进程

**结论**：

- 这一页不需要额外图片
- 以信息清晰为主

---

### 第 5 页：系统启动扩展与验证

**建议图类型**：

- 启动截图

**直接使用**：

- `docx/tex/结题报告/images/系统启动.png`

**可以在图旁边标注的关键信息**：

- `memory: ... (dtb)`
- `kmalloctest: ok`
- 进入 `$`

**结论**：

- 这页直接复用现有截图即可
- 不需要新图

---

### 第 6 页：内存系统整体框架

**建议图类型**：

- 内存系统三层结构图

**推荐方案**：

- 使用 `Mermaid` 或手工框图

**建议三层结构**：

```text
内核堆分配器：kmalloc / kmfree
虚拟内存管理：页表 / 映射 / lazy sbrk
物理页框分配：kalloc / kfree
```

**讲解目标**：

- 先让老师看到整体层次
- 后面再逐层展开

**结论**：

- 这页是结构图页，不建议用 AI 图

---

### 第 7 页：内存系统基础功能

**建议图类型**：

- 简化地址空间布局图 或 表格

**推荐方案**：

- 左边写基础能力
- 右边画简化地址空间示意：
  - code/data/bss
  - heap
  - stack
  - trapframe
  - trampoline

**结论**：

- 这一页以说明为主
- 图可以简单，不必做复杂

---

### 第 8 页：内存系统扩展部分

**建议图类型**：

- 运行时探测流程图 + `kmalloc` 结构图

**推荐方案**：

这一页可以做成左右双栏：

左栏：

```text
DTB
 -> /memory/reg
 -> phys_ram_start / phys_ram_end
 -> kinit()
```

右栏：

```text
kmalloc(n)
 -> size class slab
 -> 或大对象回退
 -> kmfree()
 -> 空 slab 页归还
```

**如果只能保留一张图**：

- 优先保留 `kmalloc/kmfree` 两级分配结构图

**结论**：

- 这页是内存部分的亮点图
- 推荐自己画，不要交给 AI 自由发挥

---

### 第 9 页：内存系统测试与验证

**建议图类型**：

- 启动日志截图

**直接使用**：

- `docx/tex/结题报告/images/kmalloctest 启动日志截图.png`

**旁边建议补的文本**：

- `128M / 256M` 探测成功
- `kmalloctest: ok`
- `pipe/file` 真实路径可用

**结论**：

- 这页截图已经足够
- 不必额外做复杂图

---

### 第 10 页：文件系统整体框架

**建议图类型**：

- 文件系统分层结构图

**推荐方案**：

- 使用 `Mermaid` 或手工框图

**建议分层**：

```text
sysfile.c
file.c
fs.c
log.c
bio.c
virtio disk
```

**讲解重点**：

- 从文件描述符一路走到底层磁盘块

**结论**：

- 这一页一定要结构清楚
- 不建议用 `imagegen`

---

### 第 11 页：文件系统基础功能

**建议图类型**：

- 功能对照表

**推荐方案**：

- 左列写老师要求
- 右列写系统当前实现

内容包括：

- 创建/删除/打开/关闭/读写
- 文件描述符表
- 路径解析
- `lseek`
- `mkfs`

**补充说明**：

- 标准输入输出重定向可以只在备注中提一句由侯奕明完成

**结论**：

- 这页不需要大图

---

### 第 12 页：文件系统扩展一

**建议图类型**：

- 日志事务图 + 双盘挂载图

**推荐方案**：

这一页做成左右双栏最合适。

左栏日志机制：

```text
begin_op
 -> log_write
 -> commit
 -> install_trans
 -> recover_from_log
```

右栏双盘结构：

```text
ROOTDEV -> /
DISK1DEV -> /disk1
```

要强调：

- 文件不能跨盘
- `/disk1` 是静态挂载点
- `..` 返回根盘
- 日志不再是简单单盘假设

**结论**：

- 这是文件系统最关键的技术图页
- 建议自己画框图

---

### 第 13 页：文件系统扩展二

**建议图类型**：

- 终端截图 + 规则表

**直接使用**：

- `docx/tex/结题报告/images/隐藏颜色截图.png`

**旁边建议列规则**：

- 目录：蓝色
- 设备文件：黄色
- 隐藏项：灰色
- `ls -a`：显示隐藏文件

**结论**：

- 这页现成截图最好用
- 不需要重新画图

---

### 第 14 页：文件系统测试与验证

**建议图类型**：

- `lseektest` 截图 + 少量命令展示

**直接使用**：

- `docx/tex/结题报告/images/lseektest截图.png`

**建议补的文字命令**：

```text

/desktop

$ ls /
desktop

$ echo hi > /disk1/x
$ cat /disk1/x
```

**结论**：

- 如果一页只能放一张图，优先保留 `lseektest`
- 双盘测试用文字列出来就够

**结论**：

- 收尾页保持干净
- 不要再堆技术图

## 六、哪些图适合用 imagegen

适合 `imagegen` 的：

- 封面图
- 过渡页背景图
- 抽象“内核系统扩展”概念图

不适合 `imagegen` 的：

- 启动流程图
- 页表结构图
- `kmalloc` 分配结构图
- 文件系统分层图
- 日志事务图
- 双盘挂载关系图

原因：

- 这些图的核心要求是结构精确，不是风格炫

## 七、最终建议

这次 PPT 配图最优策略不是“全部重新生成”，而是：

- **截图直接复用**
- **结构图自己画**
- **封面图可选 AI**

这样风险最低，也最适合课程设计答辩场景。
