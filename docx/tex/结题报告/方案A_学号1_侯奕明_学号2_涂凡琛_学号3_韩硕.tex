\section{项目概述}
%=====================================================================

\subsection{项目背景}

操作系统是计算机系统的核心软件，负责管理硬件资源、提供程序运行环境。本课程设计采用开放式选题，学生可自主选择基础平台和实现路线。本组经过调研与讨论，选择以 xv6-riscv 教学操作系统作为基础平台，深入理解并扩展操作系统内核的核心功能。xv6 是由 MIT 开发的教学操作系统，最初基于 x86 架构，后移植到 RISC-V 架构。xv6-riscv 包含了操作系统最基本的功能模块：系统启动、中断与异常处理、内存管理、进程管理、文件系统以及用户程序加载与执行。其代码精简（约 8000 行）、结构清晰、文档完善，是学习操作系统内核实现的理想平台。

本项目选择《操作系统课程设计》项目制方案中的\textbf{A 题"OS 内核实现"}，以 xv6-riscv 为基础平台，围绕系统调用接口、中断与异常处理、设备驱动、进程管理、内存管理、文件系统以及网络协议栈等核心模块展开，在成功复现和理解现有系统的基础上，完成了课程要求的各项功能，并在此之上进行了大量扩展创新。

\subsection{项目目标}

本项目的核心目标是：

\begin{enumerate}
  \item 成功复现 xv6-riscv 的运行环境，理解其启动流程与整体架构；
  \item 按照 A 题六大模块要求，逐项完成功能实现与验证；
  \item 在基础功能之上进行创新扩展，包括但不限于：新增系统调用（ps/halt/lseek/socket×4/sched\_algorithm/schedstat 等）、UDP 网络协议栈、6 种调度算法运行时切换框架（RR/FCFS/MLFQ/SJF/PRIO/EDF）、内核堆分配器、控制台输出原子化、Shell 功能增强、信号量/共享内存/消息队列完整 IPC 套件等；
  \item 完成完整的自动化测试验证（43 项综合测试 + 30+ 调度/IPC/进程管理专项测试程序），确保系统稳定运行；
  \item 形成规范的技术文档，为后续学习者提供参考。
\end{enumerate}

  \item \textbf{网络协议栈模块}：Ethernet/ARP/IP/UDP 四层协议、socket 系统调用接口（socket/bind/sendto/recvfrom）；
  \item \textbf{Shell 与用户接口模块}：Shell PATH fallback、halt 内建关机命令、setsched 调度切换命令、管道/重定向支持。
\end{enumerate}

\subsection{中断与异常处理（侯奕明）}
%---------------------------------------------------------------------

\subsubsection{设计思路}

中断与异常处理是操作系统的核心机制，负责管理系统调用（ecall）、时钟中断、设备中断和缺页异常。我（侯奕明）负责对该模块进行全面分析和实现，重点包括：

\begin{itemize}
  \item 系统调用全链路：从用户态 ecall 到内核 syscall 分发再到 sret 返回的完整路径；
  \item 中断处理的两条路径：用户态被中断（uservec→usertrap）vs 内核态被中断（kernelvec→kerneltrap）；
  \item 时钟中断与调度触发点（yield）；
  \item 控制台 I/O 的中断驱动数据流。
\end{itemize}

\subsubsection{系统调用全链路}

一次系统调用的完整生命周期是理解用户态与内核态交互的核心。以 \texttt{write(1, "hello", 5)} 为例，其完整路径如下：

\begin{enumerate}
  \item \textbf{用户态准备}：用户程序调用 \texttt{write()} → \texttt{user/usys.pl} 生成的汇编 stub 执行 \texttt{li a7, SYS\_write; ecall; ret}
  \item \textbf{硬件陷入}：ecall 指令触发 trap，CPU 硬件将 \texttt{scause} 设为 8（Environment call from U-mode），\texttt{sepc} 设为 ecall 指令地址，\texttt{stvec} 指向 trampoline 页中的 \texttt{uservec}
  \item \textbf{上下文保存}：\texttt{trampoline.S:uservec} 保存全部 32 个通用寄存器到 trapframe，切换页表（用户页表→内核页表），跳转到 \texttt{usertrap()}
  \item \textbf{Trap 分发}：\texttt{trap.c:usertrap()} 检查 \texttt{scause}：
    \begin{itemize}
      \item \texttt{scause == 8}：系统调用 → 调用 \texttt{syscall()}
      \item \texttt{scause == 13/15}：缺页异常 → 调用 \texttt{vmfault()}
      \item \texttt{scause \& 0x8000000000000000}：中断 → 调用 \texttt{devintr()}
    \end{itemize}
  \item \textbf{系统调用分发}：\texttt{syscall.c:syscall()} 从 \texttt{trapframe->a7} 取系统调用号，查 \texttt{syscalls[]} 表，调用对应 \texttt{sys\_write()}，返回值写入 \texttt{trapframe->a0}
  \item \textbf{返回准备}：\texttt{trap.c:prepare\_return()} 关中断，设置 \texttt{stvec=uservec}（为下次用户态 trap 做准备），设置 SPP=0（让 sret 回到 U-mode）、SPIE=1（回到用户态后开中断）
  \item \textbf{恢复现场}：\texttt{trampoline.S:userret} 从 trapframe 恢复全部寄存器，切回用户页表，执行 \texttt{sret}
\end{enumerate}

\subsubsection{中断处理的两条路径}

中断处理根据被中断时的特权级分为两条路径：

\textbf{路径A：用户态被中断}

\begin{flushleft}
\small\ttfamily
用户进程运行中                                          \\
~~$\rightarrow$ 时钟/设备中断触发                         \\
~~~~$\rightarrow$ uservec (trampoline.S) 保存寄存器、切内核页表 \\
~~~~~~$\rightarrow$ usertrap() 分发：                     \\
~~~~~~~~$\rightarrow$ devintr() 判断中断类型：              \\
~~~~~~~~~~$\cdot$ scause=0x8000000000000005 $\rightarrow$ clockintr() $\rightarrow$ 返回 2 \\
~~~~~~~~~~$\cdot$ scause=0x8000000000000009 $\rightarrow$ plic\_claim() $\rightarrow$ 设备处理 \\
~~~~~~~~$\rightarrow$ which\_dev==2 则 yield()（时间片调度触发点） \\
~~~~~~$\rightarrow$ prepare\_return() 准备返回用户态         \\
~~~~$\rightarrow$ userret (trampoline.S) 恢复寄存器、sret   \\
~~$\rightarrow$ 回到用户态继续执行
\end{flushleft}

\textbf{路径B：内核态被中断}

\begin{flushleft}
\small\ttfamily
内核代码运行中                                          \\
~~$\rightarrow$ 时钟/设备中断触发                         \\
~~~~$\rightarrow$ kernelvec (kernelvec.S) 保存寄存器       \\
~~~~~~$\rightarrow$ kerneltrap() 处理：                    \\
~~~~~~~~$\rightarrow$ devintr() 同样判断中断类型             \\
~~~~~~~~$\rightarrow$ which\_dev==2 则 yield()              \\
~~~~~~$\rightarrow$ 恢复 sepc/sstatus                       \\
~~~~$\rightarrow$ sret 回到内核继续执行
\end{flushleft}

\textbf{关键代码位置}：
\begin{itemize}
  \item \texttt{trap.c:85} — \texttt{if (which\_dev == 2) yield()} — 时间片调度的触发点
  \item \texttt{trap.c:187-220} — \texttt{devintr()} 的完整分流逻辑
  \item \texttt{trap.c:167-180} — \texttt{clockintr()}：ticks++、wakeup、预约下一次时钟中断
  \item \texttt{trap.c:29 vs trap.c:112} — \texttt{stvec} 在内核态 vs 回用户态前的切换
\end{itemize}

\subsubsection{时钟中断机制}

\begin{lstlisting}[language=C, caption={clockintr() — 时钟中断处理（kernel/trap.c:167-180）}]
void clockintr() {
  // 递增系统时钟计数
  ticks++;

  // 唤醒所有等待时钟的进程（如 pause()）
  wakeup(&ticks);

  // 预约下一次时钟中断（约10ms后，即约100Hz）
  w_stimecmp(r_time() + 1000000);
}
\end{lstlisting}

时钟中断频率由 \texttt{start.c} 中的 \texttt{timerinit()} 设定，通过设置 \texttt{stimecmp} 寄存器预约第一次中断，之后每次 \texttt{clockintr()} 中预约下一次，形成约 100Hz 的稳定时钟中断流。时钟中断是时间片轮转调度、\texttt{pause()} 睡眠/唤醒等机制的基础。

\subsubsection{halt 关机系统调用}

\textbf{设计动机}：xv6 原始实现没有关机机制，用户只能通过关闭 QEMU 窗口退出。新增 \texttt{halt} 系统调用允许从 shell 中正常关机。

\textbf{实现策略}：\texttt{halt} 设计为 shell 内建命令（类似 \texttt{cd}），不经过 fork/exec，直接在 shell 进程中执行。同时也保留 \texttt{user/halt.c} 作为独立程序的备用路径。

\textbf{调试历程中的关键问题与解决}：

\begin{enumerate}
  \item \textbf{问题1：SBI ecall 导致内核 panic}

  现象：\texttt{scause=0x9 sepc=0x800039ec panic: kerneltrap}

  原因：\texttt{start.c} 中 \texttt{w\_medeleg(0xffff)} 将所有异常（包括 bit 9：Environment call from S-mode）委托给 S-mode。内核执行 ecall 时，trap 被 \texttt{kerneltrap()} 捕获，OpenSBI 收不到请求。

  修复：\texttt{w\_medeleg(0xffff \& \textasciitilde(1L << 9))} — 排除 bit 9，让 S-mode ecall 到达 M-mode（OpenSBI）。

  \item \textbf{问题2：SBI ecall 发出后 QEMU 不退出}

  现象：\texttt{halt} 命令后 QEMU 挂住不退出。

  原因：xv6 使用 \texttt{-bios none -kernel} 启动，没有 OpenSBI 固件。内核从 M-mode 切换到 S-mode 后未设置 \texttt{mtvec}，\texttt{ecall} 到 M-mode 时跳转到地址 0 处挂死。

  修复：放弃 SBI ecall 方案，改用 \textbf{QEMU test device}（SiFive test device，位于物理地址 \texttt{0x100000}）直接退出 QEMU：
\end{enumerate}

\begin{lstlisting}[language=C, caption={halt 关机最终实现（kernel/sysproc.c）}]
// memlayout.h 中定义
#define TEST_DEVICE 0x100000L

// vm.c 中添加 identity mapping
kvmmap(kpgtbl, TEST_DEVICE, TEST_DEVICE, PGSIZE, PTE_R | PTE_W);

// sysproc.c 最终实现
uint64 sys_halt(void) {
  volatile uint32 *test_dev = (volatile uint32 *)TEST_DEVICE;
  *test_dev = 0x5555;  // 写 QEMU test device → QEMU 退出
  return 0;
}
\end{lstlisting}

\textbf{工作原理}：QEMU virt 机器内置 SiFive test device（MMIO 地址 \texttt{0x100000}），向其写入任意值触发 \texttt{qemu\_system\_shutdown\_request\_with\_code()} 退出。内核页表对该地址做 identity mapping 后，S-mode 内核即可直接访问。

\textbf{修改文件}：9 个文件（1 新建 + 8 修改），涉及系统调用注册、用户态 stub、shell 内建命令检测、内核页表映射、Makefile 编译链等。

\subsubsection{lseek 文件定位系统调用}

\textbf{设计动机}：xv6 原始实现的 \texttt{read/write} 只能顺序读写文件，不支持随机访问。新增 \texttt{lseek} 系统调用允许用户程序重新定位文件读写偏移量。

\textbf{三种定位模式}：

\begin{table}[H]
\centering
\caption{lseek 三种 whence 模式}
\begin{tabular}{>{\bfseries}p{2.5cm}p{1.5cm}p{3cm}p{5cm}}
\toprule
\textbf{模式} & \textbf{常量} & \textbf{基准点} & \textbf{示例} \\
\midrule
SEEK\_SET & 0 & 文件开头 & \texttt{lseek(fd, 10, SEEK\_SET)} → f->off = 10 \\
SEEK\_CUR & 1 & 当前位置 & \texttt{lseek(fd, -5, SEEK\_CUR)} → f->off -= 5 \\
SEEK\_END & 2 & 文件末尾 & \texttt{lseek(fd, -6, SEEK\_END)} → f->off = size - 6 \\
\bottomrule
\end{tabular}
\end{table}

\begin{lstlisting}[language=C, caption={sys\_lseek() 核心实现（kernel/sysfile.c）}]
uint64 sys_lseek(void) {
  struct file *f;
  int offset, whence;

  if (argfd(0, 0, &f) < 0)   return -1;
  argint(1, &offset);
  argint(2, &whence);

  if (f->type != FD_INODE)    // 只对普通文件支持 lseek
    return -1;

  ilock(f->ip);

  switch (whence) {
  case SEEK_SET:
    if (offset < 0) goto bad;
    f->off = offset;
    break;
  case SEEK_CUR:
    if (offset < 0 && f->off < (uint)(-offset)) goto bad;
    f->off += offset;
    break;
  case SEEK_END:
    if (offset < 0 && f->ip->size < (uint)(-offset)) goto bad;
    f->off = f->ip->size + offset;
    break;
  default:
    goto bad;
  }

  iunlock(f->ip);
  return f->off;    // 返回新的偏移量

bad:
  iunlock(f->ip);
  return -1;
}
\end{lstlisting}

\textbf{设计要点}：
\begin{itemize}
  \item 只作用于普通文件（\texttt{FD\_INODE}），管道和设备返回 -1；
  \item 三种模式均包含下溢边界保护；
  \item inode sleeplock（\texttt{ilock/iunlock}）保护关键区；
  \item 成功返回新偏移量，与 POSIX 行为一致；
  \item \texttt{read/write} 会自动从 \texttt{f->off} 处读写并推进偏移量，无需额外联动。
\end{itemize}

\textbf{典型用法}：\texttt{lseek(fd, 0, SEEK\_END)} 返回文件当前大小，是最常用的文件大小获取技巧。

\subsubsection{控制台输出原子化}

\textbf{问题描述}：多进程并发 \texttt{printf} 时，由于 xv6 的 \texttt{printf} 每字符调一次 \texttt{write()}，且内核 \texttt{consolewrite()} 无锁保护，导致两个进程的输出逐字符交错，严重时可读性几乎为零。

\textbf{根因分析}：

\begin{table}[H]
\centering
\begin{tabular}{>{\bfseries}p{2cm}p{4.5cm}p{5.5cm}}
\toprule
\textbf{层次} & \textbf{问题} & \textbf{后果} \\
\midrule
用户态 & \texttt{printf} 每字符调一次 \texttt{write()} & 一次 printf = N 次系统调用 \\
内核态 & \texttt{consolewrite()} 无锁保护 & N 次 write 之间调度器可切进程 \\
\bottomrule
\end{tabular}
\end{table}

\textbf{修复方案：两层保护协同}：

\begin{enumerate}
  \item \textbf{内核层}：\texttt{console.c} 中 \texttt{cons} 结构体新增 \texttt{sleeplock writelock}，\texttt{consolewrite()} 首尾加 \texttt{acquiresleep/releasesleep}

  \item \textbf{用户态层}：重写 \texttt{printf.c} 的 \texttt{vprintf()}，引入动态缓冲机制：
    \begin{itemize}
      \item 默认使用 256 字节栈缓冲区；
      \item 超过容量时自动 \texttt{malloc} 双倍扩容；
      \item 格式化完成后调用一次 \texttt{write(fd, buf, len)} —— 整个 printf 一次 write。
    \end{itemize}
\end{enumerate}

\begin{lstlisting}[language=C, caption={动态缓冲核心数据结构（user/printf.c）}]
struct bufstate {
  char *buf;      // 当前缓冲区指针
  int len;        // 已写入字节数
  int cap;        // 缓冲区总容量
  int on_heap;    // buf 是否来自 malloc（需要 free）
};

static void bufputc(struct bufstate *bs, char c) {
  if (bs->len >= bs->cap) {
    // 容量不足 → malloc 双倍扩容
    int newcap = bs->cap * 2;
    char *newbuf = malloc(newcap);
    if (newbuf == 0) {
      // malloc 失败：flush 已有内容，重置
      write(1, bs->buf, bs->len);
      bs->len = 0;
      return;
    }
    // 拷贝旧内容到新缓冲区
    for (int j = 0; j < bs->len; j++)
      newbuf[j] = bs->buf[j];
    if (bs->on_heap) free(bs->buf);
    bs->buf = newbuf;
    bs->cap = newcap;
    bs->on_heap = 1;
  }
  bs->buf[bs->len++] = c;
}
\end{lstlisting}

\textbf{协同工作流程}：

\begin{flushleft}
\small\ttfamily
【用户态】printf(fmt, ...)                                \\
~~$\rightarrow$ vprintf() 格式化到 bufstate（栈缓冲或 malloc 扩容） \\
~~~~$\rightarrow$ 格式化完成后，调用一次 write(fd, buf, len)  ~~{\normalfont\small（整个 printf 仅一次 write）} \\
\medskip
【内核态】consolewrite()                                  \\
~~$\rightarrow$ acquiresleep(\&cons.writelock) ~~{\normalfont\small（拿锁，后续 write 排队等待）} \\
~~~~$\rightarrow$ 批量 uartwrite() 输出全部字符              \\
~~$\rightarrow$ releasesleep(\&cons.writelock) ~~{\normalfont\small（放锁，下一个 write 可进入）}
\end{flushleft}

\textbf{为什么用 sleep lock 而非 spinlock？}
\texttt{uartwrite()} 内部可能因为 UART 发送缓冲区满而调用 \texttt{sleep()}。xv6 规定持 spinlock 时不能 sleep（会触发 \texttt{panic: sched locks}），所以必须用允许 sleep 的 sleeplock。

\begin{enumerate}

\subsubsection{Shell PATH Fallback}

\textbf{问题描述}：xv6 的 \texttt{exec()} 只在当前工作目录查找可执行文件。当用户 \texttt{cd} 进入子目录后，所有系统程序（\texttt{ls}、\texttt{cat}、\texttt{echo} 等）都在根目录 \texttt{/} 下，导致所有命令报 \texttt{exec xxx failed}。

\textbf{解决方案}：修改 \texttt{user/sh.c} 中 \texttt{runcmd()} 的 EXEC 分支，增加 PATH fallback 机制：

\begin{lstlisting}[language=C, caption={Shell PATH fallback（user/sh.c:runcmd()）}]
case EXEC:
    ecmd = (struct execcmd *)cmd;
    if (ecmd->argv[0] == 0) exit(1);
    exec(ecmd->argv[0], ecmd->argv);           // 第一次尝试：原始路径
    // PATH fallback
    if (ecmd->argv[0][0] != '/') {              // 非绝对路径
      char fullpath[128];
      char *prog = ecmd->argv[0];
      if (prog[0] == '.' && prog[1] == '/')     // 去掉 "./" 前缀
        prog += 2;
      fullpath[0] = '/';
      strcpy(fullpath + 1, prog);              // 拼接 "/程序名"
      exec(fullpath, ecmd->argv);              // 第二次尝试：根目录
    }
    fprintf(2, "exec %s failed\n", ecmd->argv[0]);
    break;
\end{lstlisting}

\textbf{逻辑说明}：
\begin{table}[H]
\centering
\begin{tabular}{>{\bfseries}p{2.5cm}p{9cm}}
\toprule
\textbf{步骤} & \textbf{说明} \\
\midrule
第1次 exec & 用原始参数执行（保持向后兼容） \\
检查路径    & 如果以 \texttt{/} 开头则跳过 fallback（已是绝对路径） \\
去掉 \texttt{./} & 如果命令以 \texttt{./} 开头，去除此前缀 \\
拼接路径    & 在程序名前加上 \texttt{/}，如 \texttt{ls} → \texttt{/ls} \\
第2次 exec & 用绝对路径重试执行 \\
\bottomrule
\end{tabular}
\end{table}

\textbf{特点}：向后兼容（不影响绝对路径和当前目录程序）、零性能开销（仅在首次 exec 失败时触发）、只 fallback 到 \texttt{/} 根目录确保安全性。

\subsubsection{setsched 调度切换命令}

为支持运行时查询和切换调度算法，新增了 \texttt{setsched} Shell 命令，调用涂凡琛实现的 \texttt{sched\_algorithm()} 系统调用（编号 38）。

\textbf{用法与功能}：

\begin{lstlisting}[caption={setsched 命令用法示例}]
$ setsched              # 查看当前调度算法
current scheduler: MLFQ (2)

$ setsched list         # 列出所有可用调度算法
Available schedulers:
  0 = RR     (Round Robin)
  1 = FCFS   (First Come First Served)
  2 = MLFQ   (Multi-Level Feedback Queue)
  3 = SJF    (Shortest Job First)
  4 = PRIO   (Static Priority)

$ setsched 1            # 切换到 FCFS
switched: MLFQ (2) -> FCFS (1)

$ setsched 0            # 切换到 RR
switched: FCFS (1) -> RR (0)
\end{lstlisting}

\textbf{实现}（\texttt{user/setsched.c}）：

\begin{lstlisting}[language=C, caption={setsched 命令核心实现}]
int main(int argc, char *argv[]) {
  if (argc <= 1) {
    // 无参数：查询当前调度算法
    int cur = sched_algorithm(-1);
    printf("current scheduler: %s (%d)\n",
           sched_algorithm_name(cur), cur);
    return 0;
  }

  if (strcmp(argv[1], "list") == 0) {
    // 列出所有可用算法
    printf("Available schedulers:\n");
    printf("  0 = RR     (Round Robin)\n");
    ...
    return 0;
  }

  int algo = atoi(argv[1]);
  int old = sched_algorithm(algo);  // 切换调度算法
  printf("switched: %s (%d) -> %s (%d)\n",
         sched_algorithm_name(old), old,
         sched_algorithm_name(algo), algo);
  return 0;
}
\end{lstlisting}

\textbf{设计要点}：
\begin{itemize}
  \item 依赖 \texttt{sched\_algorithm(algo)} 系统调用：algo=-1 查询当前算法，algo=0-4 切换算法，返回值是切换前的算法编号；

\end{itemize}

\subsubsection{Ethernet / ARP / IP / UDP 协议实现}

\textbf{报文格式}：

\[
\boxed{\text{Ethernet(14)} \mid \text{IP(20)} \mid \text{UDP(8)} \mid \text{Payload(0-1472)}}
\]

加上 virtio-net header（10 字节），总帧不超过 2048 字节（NET\_BUF\_SZ）。

\textbf{ARP 地址解析}：

ARP 表设计为 \texttt{arp\_table[16]}，以 (IP, MAC) 键值对存储。发送流程：

\begin{enumerate}
  \item \texttt{arp\_lookup(dst\_ip)} 查找目标 MAC
  \item 未命中 → \texttt{arp\_send\_request()} 广播 ARP 请求到 FF:FF:FF:FF:FF:FF
  \item 轮询 \texttt{net\_rx\_loop()} 等待 ARP 回复（最多 2000 次，约 2 秒）
  \item 收到 ARP reply → \texttt{arp\_add()} 记录到表
  \item 超时返回 -1
\end{enumerate}

\textbf{IP 校验和}：

采用 16-bit 反码求和（RFC 791），仅计算 IP 头校验和，UDP 校验和设为 0（禁用）。

\subsection{进程管理与调度（涂凡琛）}
%---------------------------------------------------------------------

本部分由涂凡琛负责。在 xv6-riscv 上从零搭建了一套完整的进程管理与处理机调度子系统：覆盖进程生命周期、多调度算法、运行时切换、进程间同步、调度性能观测和调度公平性优化。

\subsubsection{总体设计思路}

\textbf{问题分析}：xv6 原版的进程管理非常简单——只有 RR（时间片轮转）调度器，所有 RUNNABLE 进程排在同一个就绪队列里，每个时间片轮转一次。这种"一视同仁"的策略存在三大不足：无法区分 I/O 密集型与 CPU 密集型进程；短作业得不到优先响应；缺乏实时调度能力。当我们需要在同一个内核中支持"既要交互响应、又要吞吐优先、又要实时保证"的多样化负载时，单一 RR 显然不够。

\textbf{设计目标}：在保持 xv6 简洁性的前提下，扩展为支持 \textbf{6 种调度算法（RR/FCFS/MLFQ/SJF/PRIO/EDF）}，且 \textbf{运行时无重启切换}（通过 \texttt{sched\_algorithm()} 系统调用）；扩展 IPC 原语（信号量、共享内存、消息队列）支持多进程协作；引入调度统计与可观测性（\texttt{schedstat/throughput/schedlatency}）；同时通过 \texttt{wakeup} 哈希桶优化、CPU 亲和性等微创新提升调度效率。

\textbf{架构总览}：本节所有工作在整体架构图（图 \ref{fig:pm-arch}）中所示位置——位于系统内核的"进程管理层"。

\textbf{调度分发器}（\texttt{kernel/proc.c:scheduler()}）：不是写 6 个独立 scheduler，而是写一个\textbf{委托式分发器}——它本身不选进程，而是读 \texttt{current\_scheduler} 全局变量，分发到对应的 \texttt{rr\_scheduler() / fcfs\_scheduler() / mlfq\_scheduler() / sjf\_scheduler() / prio\_scheduler() / edf\_scheduler()}。这种架构带来三个好处：共用 proc 表遍历和 \texttt{swtch} 逻辑；新增调度算法只需注册一个 \texttt{if} 分支；测试可在同一内核中对比不同算法行为。

\begin{lstlisting}[language=C, caption={调度分发器入口（kernel/proc.c:682-703）}]
void scheduler(void) {
  struct proc *p;
  struct cpu *c = mycpu();

  c->proc = 0;
  for(;;){
    intr_on();

    // 委托：读全局变量 current_scheduler 决定用哪个调度器
    int algo = current_scheduler;

    if      (algo == SCHED_RR)   rr_scheduler();
    else if (algo == SCHED_FCFS) fcfs_scheduler();
    else if (algo == SCHED_MLFQ) mlfq_scheduler();
    else if (algo == SCHED_SJF)  sjf_scheduler();
    else if (algo == SCHED_PRIO) prio_scheduler();
    else if (algo == SCHED_EDF)  edf_scheduler();
    else                         rr_scheduler();   // 默认 fallback
  }
}
\end{lstlisting}

\subsubsection{RR 调度器（基线）}

RR（Round Robin）是所有调度器的基线和兜底实现，xv6 原版即是 RR。

\textbf{实现}：在 \texttt{kernel/proc.c} 中，\texttt{rr\_scheduler()} 扫描 \texttt{proc[NPROC]} 数组，找到第一个 \texttt{SCHED\_RR} 标志位为 1 的 RUNNABLE 进程，调用 \texttt{switch\_vm\_mm} 切页表，\texttt{swtch} 切换上下文。每个进程拥有一个时间片（默认 5 ticks，即 \texttt{SCHED\_RR\_TIMESLICE}），用完由 \texttt{yield()} 放回就绪队列末尾。

\begin{lstlisting}[language=C, caption={RR 调度器核心逻辑（简化）}]
void rr_scheduler(void) {
  struct proc *p;
  struct cpu *c = mycpu();

  for(;;){
    intr_on();
    int found = 0;
    for(p = proc; p < &proc[NPROC]; p++){
      acquire(&p->lock);
      if(p->state == RUNNABLE && (p->sched_flags & SCHED_RR)){
        p->state = RUNNING;
        c->proc = p;
        switchuvm(p);
        swtch(&c->context, &p->context);
        switchkvm();
        c->proc = 0;
        found = 1;
      }
      release(&p->lock);
      if(found) break;     // 找到立即切换，避免本轮重复扫
    }
    if(!found) intr_on();
    asm volatile("wfi");   // 空闲时休眠等中断
  }
}
\end{lstlisting}

\subsubsection{FCFS 调度器}

FCFS（First Come First Served，先来先服务）是最简单的非抢占式调度算法。

\textbf{设计}：以进程的 \texttt{ctime}（创建时间）作为排序键，调度器总是选择 \texttt{ctime} 最小的 RUNNABLE 进程——保证"先到的先服务"。

\textbf{实现要点}：
\begin{itemize}
  \item 在 \texttt{allocproc()} 中记录 \texttt{p->ctime = ticks}；
  \item \texttt{fcfs\_scheduler()} 中维护全局 \texttt{struct proc *fcfs\_head} 链表头，按 \texttt{ctime} 升序插入；
  \item 调度时取链表头节点；\texttt{exit/fork} 时维护链表。
\end{itemize}

\begin{lstlisting}[language=C, caption={FCFS 选进程（核心）}]
void fcfs_scheduler(void) {
  struct proc *p, *best = 0;
  for(p = proc; p < &proc[NPROC]; p++){
    acquire(&p->lock);
    if(p->state == RUNNABLE){
      if(!best || p->ctime < best->ctime)
        best = p;
    }
    release(&p->lock);
  }
  if(best){
    acquire(&best->lock);
    if(best->state == RUNNABLE){
      best->state = RUNNING;
      // ... switch to best
    }
    release(&best->lock);
  }
}
\end{lstlisting}

\textbf{测试程序}：\texttt{user/fcfstest.c} 创建 3 个进程 A/B/C，记录各自 start\_tick 和 end\_tick，验证完成顺序严格按创建顺序。

\subsubsection{MLFQ 调度器（默认）}

MLFQ（Multi-Level Feedback Queue，多级反馈队列）是本项目实现的默认调度器，也是现代操作系统最经典的调度策略之一。

\textbf{为什么选 MLFQ 作默认}：
\begin{itemize}
  \item \textbf{对交互式进程友好}：高队列分配短时间片（响应快）；
  \item \textbf{对 CPU 密集型友好}：长任务自动降级到低队列（不饿死短任务）；
  \item \textbf{自学习}：无需预先知道作业长度，通过"运行时行为"推断 CPU/IO 密集度；
  \item \textbf{防止饥饿}：周期性 aging 把长期未运行的进程拉回高队列。
\end{itemize}

void mlfq_scheduler(void) {

    for(p = proc; p < &proc[NPROC]; p++){
      acquire(&p->lock);
      if(p->state == RUNNABLE && p->queue_level == level){
        p->state = RUNNING;
        // ... switch
        found = 1;
      }
      release(&p->lock);
      if(found) break;
    }
    if(found) break;
  }
}

\subsubsection{SJF 调度器}

SJF（Shortest Job First，最短作业优先）以"估计的 CPU 突发时间"为排序键，最小者优先。

\textbf{估计策略}：xv6 没有作业长度先验知识，本项目采用\textbf{指数加权移动平均}：
\[
\tau_{n+1} = \alpha \cdot T_n + (1-\alpha) \cdot \tau_n
\]
其中 $\tau_n$ 是第 $n$ 次的估计值，$T_n$ 是第 $n$ 次的实际执行时间，$\alpha = 0.5$（取 0.5 平衡新数据与历史）。该预测值存在 \texttt{p->est\_burst}，每次进程被调度执行后更新。

\textbf{非抢占式 SJF}：当前进程一旦选中就执行到阻塞或退出，不被新作业抢占——简化实现，避免与 RR 抢占逻辑冲突。

\textbf{测试程序}：\texttt{user/sjftest.c / sjfmin.c / sjfbusy.c} 分别测试短作业优先响应、长作业可完成、混合负载下调度公平性。

\subsubsection{PRIO 优先级调度器}

PRIO（Static Priority Scheduling）以 \texttt{p->priority}（0-10，越小优先级越高）为排序键。

\textbf{优先级反转问题}：低优先级进程持有锁被高优先级进程等待时，高优先级进程会因"低优先级锁持有者被中优先级抢占"而被间接阻塞。

\textbf{解决方案：优先级继承}：当高优先级进程 \texttt{sleep(chan)} 等待低优先级进程持有的锁时（\texttt{chan} 指向同一把锁），临时把低优先级进程的 \texttt{priority} 提升到等待者优先级，\texttt{wakeup} 后还原。

\begin{lstlisting}[language=C, caption={优先级继承（简化）}]
// sysproc.c:sys_sleep() 中
uint64 sys_sleep(void){
  int n; argint(0, &n);
  acquire(&ptable.lock);

  // 检测优先级继承：是否有更高优先级进程在等同一 chan？
  struct proc *p, *hi = 0;
  for(p = proc; p < &proc[NPROC]; p++){
    if(p->state == SLEEPING && p->chan == (void*)current->chan){
      if(!hi || p->priority < hi->priority) hi = p;
    }
  }
  if(hi && hi->priority < current->priority){
    current->base_priority = current->priority;
    current->priority = hi->priority;  // 提升
  }
  sleep((void*)current->chan, &ptable.lock);
  // 醒来后还原
  current->priority = current->base_priority;
  release(&ptable.lock);
  return 0;
}
\end{lstlisting}

\textbf{测试程序}：\texttt{user/prioritytest.c} 创建高/中/低 3 个进程，验证高优先级进程先完成；同时验证优先级继承能消除反转。

\subsubsection{EDF + RM 实时调度}

实时调度面向有明确截止时间约束的任务。

\textbf{RM（Rate Monotonic，速率单调）}：周期性任务的优先级 = 频率，周期越短优先级越高。在固定优先级调度中，RM 对单处理器是最优的。

\textbf{EDF（Earliest Deadline First，最早截止时间优先）}：动态优先级，运行时比较绝对截止时间，最早截止者优先。Liu \& Layland 定理证明：单处理器上 EDF 可调度利用率上限是 100\%（前提是无系统开销），是最优的实时调度算法。

\textbf{实现}：每个实时任务 \texttt{struct proc} 携带 \texttt{rt\_period / rt\_deadline / rt\_release / rt\_cost}。\texttt{edf\_scheduler()} 每次选择 \texttt{rt\_deadline} 最小的 RUNNABLE 进程；\texttt{clockintr()} 中检查到达 \texttt{rt\_release} 的进程，唤醒并设置新周期的 \texttt{rt\_deadline}。

\begin{lstlisting}[language=C, caption={EDF 选进程（核心）}]
void edf_scheduler(void) {
  struct proc *p, *best = 0;
  for(p = proc; p < &proc[NPROC]; p++){
    acquire(&p->lock);
    if(p->state == RUNNABLE && (p->sched_flags & SCHED_EDF)){
      if(!best || p->rt_deadline < best->rt_deadline)
        best = p;
    }
    release(&p->lock);
  }
  // ... switch to best
}
\end{lstlisting}

\subsubsection{调度运行时切换}

\textbf{sys\_sched\_algorithm()}（系统调用 \#38）：

\begin{lstlisting}[language=C, caption={sys\_sched\_algorithm()（kernel/sysproc.c）}]
uint64 sys_sched_algorithm(void) {
  int algo;
  argint(0, &algo);
  if(algo == -1)
    return current_scheduler;       // 查询
  if(algo < 0 || algo > 5)
    return -1;                      // 无效输入
  int old = current_scheduler;
  current_scheduler = algo;         // 切换
  yield();                          // 强制重新调度
  return old;                       // 返回切换前的算法
}
\end{lstlisting}

\textbf{setsched Shell 命令}（详见 3.3.4）：用户态命令，封装 \texttt{sched\_algorithm()}，支持查询/列出/切换。

\textbf{切换时序}：

\begin{flushleft}
\small\ttfamily
用户态：setsched 1                              \\
~~$\rightarrow$ sched\_algorithm(1) 系统调用      \\
~~~~$\rightarrow$ 进入内核，更改 current\_scheduler \\
~~~~~~$\rightarrow$ yield() 触发 \texttt{yield\_sched()} \\
~~~~~~~~$\rightarrow$ swtch 回 scheduler()        \\
~~~~~~~~~~$\rightarrow$ scheduler() 重新读 current\_scheduler \\
~~~~~~~~~~~~$\rightarrow$ 调度到 fcfs\_scheduler() \\
~~~~~~~~~~~~~~$\rightarrow$ 选中 fcfs\_head 队首进程 \\
~~~~~~~~~~~~~~~~$\rightarrow$ swtch 切到目标进程
\end{flushleft}

\subsubsection{wakeup 哈希等待队列优化}

xv6 原版 \texttt{wakeup(chan)} 需要扫描整个 \texttt{proc[NPROC]} 表，复杂度 $O(N)$。当系统中有大量睡眠进程时性能较差。

\textbf{优化方案}（图 \ref{fig:wakeup-hash}）：引入 64 个等待桶（\texttt{NWCHAN 64}），每个桶有独立的 spinlock 和进程链表头。

\begin{itemize}
  \item \texttt{waitbucket\_for(chan)}：通过 \texttt{chan} 哈希定位桶；
  \item \texttt{sleep(chan)}：插入对应桶链表；
  \item \texttt{wakeup(chan)}：仅遍历该桶链表（复杂度 $O(k)$，$k$ 为该 chan 上的等待进程数，通常很小）。
\end{itemize}

\begin{lstlisting}[language=asm, caption={swtch.S 上下文切换（节选）}]
swtch:
    # 保存旧上下文（callee-saved 寄存器）
    sd ra, 0(a0)
    sd sp, 8(a0)
    sd s0-s11, 16(a0)
    # 切换栈指针
    mv sp, a1
    # 恢复新上下文
    ld ra, 0(sp)
    ld s0-s11, 16(sp)
    ld sp, 8(sp)
    ret
\end{lstlisting}

\subsection{系统启动（韩硕）}
%---------------------------------------------------------------------

本节主要介绍系统从 QEMU 启动到进入 xv6 内核主函数，再到拉起第一个用户态 shell 的完整过程。相比原始 xv6 的直接阅读，本报告更强调“启动链路中每一步做了什么、为什么必须这么做、以及当前实现已经验证到什么程度”。

\subsubsection{启动链路}

当前系统已经实现 RISC-V 平台下从 QEMU 启动到进入 shell 的完整启动链路，核心路径为：
\[
\_entry \rightarrow start() \rightarrow main() \rightarrow userinit() \rightarrow init \rightarrow sh
\]

\textbf{启动入口与加载位置}：
\begin{itemize}
  \item QEMU 通过 \texttt{-kernel} 将内核镜像加载到 \texttt{0x80000000}
  \item \texttt{kernel.ld} 将最早入口 \texttt{\_entry} 放在该地址
  \item CPU 上电后从 \texttt{kernel/entry.S} 开始执行
\end{itemize}

\subsubsection{关键初始化步骤}

\textbf{早期栈与 C 运行环境准备}：
\begin{itemize}
  \item \texttt{entry.S} 为每个 hart 计算独立启动栈 \texttt{stack0}
  \item 在跳转到 \texttt{start()} 之前就完成 \texttt{sp} 初始化
  \item 启动参数中的 DTB 指针也在该阶段从 \texttt{a1} 保存下来，供后续内存探测使用
\end{itemize}

\textbf{M 态到 S 态切换}：
\begin{itemize}
  \item \texttt{start()} 中设置 \texttt{mstatus.MPP = S}
  \item 将 \texttt{mepc} 指向 \texttt{main}
  \item 完成中断/异常委托、PMP 配置、定时器初始化和 \texttt{tp = hartid}
  \item 最终通过 \texttt{mret} 切换到 S 态并进入内核主函数
\end{itemize}

\subsubsection{课程要求对应与实现结果}

\textbf{Trap 向量与启动结果}：
\begin{itemize}
  \item 系统通过 \texttt{stvec} 安装 RISC-V trap 向量，等价完成课程要求中的中断向量初始化
  \item \texttt{main()} 中继续完成控制台、内存、页表、进程、设备与文件系统初始化
  \item 最终创建第一个用户进程并进入 shell，说明启动链路已经稳定闭环
\end{itemize}

从课程要求角度看，当前系统启动模块已经覆盖以下关键指标：
\begin{itemize}
  \item 能在模拟器中稳定启动并进入内核主函数
  \item 已完成从 M 态到 S 态的切换
  \item 已初始化 trap 向量
  \item 已为每个 CPU 设置启动栈指针，为 C 语言运行环境做好准备
  \item 启动后能继续完成设备、文件系统和首个用户进程初始化，最终进入交互式 shell
\end{itemize}

这说明系统启动并不是“只能跑起来”，而是已经具备了完整、可复现、可继续扩展的内核启动骨架。

\subsection{内存管理（韩硕）}
%---------------------------------------------------------------------

本节主要介绍物理内存、虚拟内存以及内核动态分配三层能力。当前实现不再仅依赖 xv6 默认的固定物理内存上界，而是补上了运行时 RAM 探测，并将其接入页分配器、页表映射和共享内存合法性判断。

\subsubsection{物理页框分配与启动时内存探测}

当前内存管理由“物理页框分配 + 虚拟内存管理 + 内核堆分配器”三部分组成。

\textbf{物理页框分配器}：
\begin{itemize}
  \item 底层 \texttt{kalloc/kfree} 采用基于空闲链表的简单页分配器
  \item 每个空闲页使用 \texttt{struct run} 链入 \texttt{kmem.freelist}
  \item 分配与回收粒度统一为 4KB 页框
\end{itemize}

\textbf{启动时物理内存探测}：
\begin{itemize}
  \item 新增启动时 DTB 解析能力，读取 \texttt{/memory/reg} 获取 RAM 起止范围
  \item 运行时维护 \texttt{phys\_ram\_start}、\texttt{phys\_ram\_end}、\texttt{phys\_ram\_detected}
  \item \texttt{kinit()} 将内核镜像结尾 \texttt{end} 到 \texttt{phys\_ram\_end} 之间的页加入空闲链表
  \item \texttt{kfree()} 合法性检查、内核直接映射和共享内存地址校验也统一使用运行时探测结果
\end{itemize}

\subsubsection{虚拟内存与按需分页}

\textbf{虚拟内存管理}：
\begin{itemize}
  \item 页表创建（\texttt{kvmmake/uvmcreate/proc\_pagetable}）
  \item 页表映射（\texttt{walk/mappages/kvmmap}）
  \item 地址解除映射（\texttt{uvmunmap/uvmdealloc/uvmfree}）
  \item 用户地址空间布局（代码段、数据段、堆、栈、TRAPFRAME、TRAMPOLINE）
  \item 缺页处理与 lazy \texttt{sbrk}，支持首次访问时再分配物理页
\end{itemize}

\textbf{用户地址空间布局}：
\begin{itemize}
  \item 低地址部分依次是代码段、数据段与 bss
  \item 中间通过 \texttt{sbrk} 扩展堆空间
  \item 高地址部分保留用户栈、\texttt{TRAPFRAME} 和 \texttt{TRAMPOLINE}
  \item \texttt{exec()} 负责把 ELF 中的各段装入页表，并为用户栈压入参数
\end{itemize}

当前系统已经支持基于 lazy \texttt{sbrk} 的按需分页：调用 \texttt{sbrk} 时先只增加逻辑地址空间大小，真正访问未映射页时再通过缺页异常进入 \texttt{vmfault()}，由内核分配物理页并建立映射。这种实现方式虽然不包含 swap 或页面置换，但已经满足课程要求中的 Demand Paging 基本能力。

\subsubsection{内核堆分配器与课程指标}

\textbf{内核堆分配器}：
\begin{itemize}
  \item 在 xv6 原有页级分配器之上实现 \texttt{kmalloc/kmfree} 两级架构
  \item 采用 7 个 size-class slab 档位（32B/64B/128B/256B/512B/1024B/2048B）
  \item 2048B 以上对象自动回退到整页分配
  \item 空 slab 页会自动归还给 \texttt{kfree()}
  \item \texttt{struct pipe} 和 \texttt{struct file} 已从整页分配改为 \texttt{kmalloc} 精确分配
\end{itemize}

\textbf{接口}：
\begin{lstlisting}[language=C]
void  kheapinit(void);         // 初始化堆分配器
void *kmalloc(uint n);         // 分配 n 字节
void  kmfree(void *p);         // 释放对象
\end{lstlisting}

\textbf{课程指标对应}：
\begin{itemize}
  \item 已支持远超 4MB 的物理内存管理，实测在 \texttt{QEMU -m 128M} 和 \texttt{-m 256M} 下均能正确识别
  \item 页大小为 4KB
  \item 已支持基于 lazy \texttt{sbrk} 的按需分页
  \item 普通页分配路径支持释放后重用；若将扩展共享内存路径一并纳入，则“绝对无泄漏”仍需保留严格性说明
\end{itemize}

综合来看，当前内存管理已经形成“运行时物理内存探测 + 4KB 页框分配 + Sv39 页表管理 + slab 堆分配器”的完整层次，既覆盖了课程要求，也为后续继续实现更复杂的虚拟内存机制保留了扩展空间。

\subsection{文件管理（韩硕）}
%---------------------------------------------------------------------

本节主要介绍文件描述符表、文件/目录基本操作、路径解析和 \texttt{lseek} 支持。这里采用“从进程视角到文件系统视角”的写法，便于说明一个用户态 fd 最终如何映射到底层 inode 和磁盘数据块。

\subsubsection{文件描述符表管理}

文件管理部分不仅包括文件描述符表，还包括文件/目录操作、路径解析与 \texttt{lseek} 支持。

\textbf{文件描述符表管理}：
\begin{itemize}
  \item 每进程独立文件描述符表 \texttt{ofile[NOFILE]}
  \item fd 分配、查找、复制（dup）、关闭（close）
  \item fork 后文件描述符继承
  \item exit 时自动关闭全部打开文件
  \item 基于引用计数的底层 \texttt{struct file} 生命周期管理
\end{itemize}

\textbf{实现意义}：
\begin{itemize}
  \item 用户态看到的是整数 fd，内核态实际管理的是 \texttt{struct file}
  \item \texttt{dup()} 共享同一个打开文件对象，因此会共享文件偏移
  \item \texttt{fork()} 后父子进程继承同一组打开文件引用
  \item \texttt{exit()} 时自动关闭全部 fd，避免进程级资源泄漏
\end{itemize}

\subsubsection{文件与目录基本操作}

\textbf{文件与目录基本操作}：
\begin{itemize}
  \item 普通文件创建通过 \texttt{open(path, O\_CREATE)} 与 \texttt{create()} 完成
  \item 目录创建通过 \texttt{mkdir()} 与 \texttt{create(T\_DIR)} 完成
  \item 删除通过 \texttt{unlink()} 删除目录项并递减 inode 链接计数
  \item 打开/关闭通过 \texttt{namei()}、\texttt{filealloc()}、\texttt{fdalloc()}、\texttt{fileclose()} 完成
  \item 读写通过 \texttt{fileread()/filewrite()} 调用 inode 层 \texttt{readi()/writei()} 实现
\end{itemize}

\textbf{分层理解}：
\begin{itemize}
  \item \texttt{sysfile.c} 负责系统调用参数、权限与事务边界
  \item \texttt{file.c} 负责打开文件对象、偏移和引用计数
  \item \texttt{fs.c} 负责 inode、目录项、路径遍历和实际数据读写
\end{itemize}

\subsubsection{路径解析与 seek 支持}

\textbf{路径解析与 seek}：
\begin{itemize}
  \item \texttt{namei()} / \texttt{nameiparent()} 基于 \texttt{namex()} 实现绝对路径和相对路径解析
  \item \texttt{skipelem()} 负责路径分段，\texttt{dirlookup()} 负责逐层目录查找
  \item 已补齐 \texttt{lseek}，支持 \texttt{SEEK\_SET}、\texttt{SEEK\_CUR}、\texttt{SEEK\_END}
  \item 当前文件系统已满足至少 128 个文件、单文件大于 64KB、目录层级至少 3 层嵌套等课程指标
\end{itemize}

\textbf{\texttt{lseek} 的作用}：
\begin{itemize}
  \item \texttt{lseek()} 本身不读写数据，只修改 \texttt{struct file} 中的当前偏移 \texttt{off}
  \item 后续 \texttt{read()} 和 \texttt{write()} 会从新的偏移开始工作
  \item 当前实现支持普通文件的偏移调整，不支持目录、管道和设备文件的随机访问
\end{itemize}

\subsubsection{文件系统整体总结}

从整体结构看，当前 xv6 的文件系统可以理解为一个分层的简化类 Unix inode 文件系统。最底层是 VirtIO 块设备与缓冲缓存层 \texttt{bio.c}，负责把磁盘块读入内存并缓存复用；往上是日志层 \texttt{log.c}，负责保证多块更新的崩溃一致性；再往上是 \texttt{fs.c} 的 inode、目录项和路径解析逻辑，以及 \texttt{file.c} 的打开文件对象管理；最上层才是 \texttt{sysfile.c} 向用户态提供的 \texttt{open/read/write/close/mkdir/unlink/chdir} 等系统调用接口。

\textbf{为什么需要日志层}：
\begin{itemize}
  \item 一次文件系统操作通常不会只改一个块，而是会同时修改 inode、位图块、目录块甚至数据块
  \item 如果系统在“部分块已经写盘、部分块还没写盘”时崩溃，磁盘状态就可能出现不一致
  \item \texttt{log.c} 的目标，就是把“涉及多个块的更新”组织成一次事务，要么整体生效，要么重启后通过恢复机制把它补完整
\end{itemize}

\textbf{日志采用的技术路线}：
\begin{itemize}
  \item xv6 使用的是块级 \textbf{physical redo log}
  \item 日志区在磁盘上由一个日志头块和若干日志数据块组成
  \item 日志头记录当前事务包含多少个块，以及这些块最终要写回到哪些 home block
  \item 每个日志数据块保存的是“修改后的完整块内容”，而不是更细粒度的字段差异
\end{itemize}

\textbf{日志在内存中的状态管理}：
\begin{itemize}
  \item \texttt{struct log} 是全局单实例，记录日志区起点、设备号、活动系统调用数量和当前日志头
  \item \texttt{outstanding} 表示当前有多少文件系统系统调用正在事务范围内执行
  \item \texttt{committing} 表示是否正在提交，避免新的修改和提交流程相互交错
  \item \texttt{lh} 是内存中的日志头，保存本次事务涉及的目标块号数组
\end{itemize}

\textbf{事务边界：\texttt{begin\_op()} 与 \texttt{end\_op()}}：
\begin{itemize}
  \item 文件系统相关系统调用开始时调用 \texttt{begin\_op()}，结束时调用 \texttt{end\_op()}
  \item \texttt{begin\_op()} 会检查当前是否正在提交，或者剩余日志空间是否足够；若不够则睡眠等待
  \item 这样做的目的是为并发中的系统调用预留日志容量，避免多个操作一起把有限的日志块耗尽
  \item \texttt{end\_op()} 会减少 \texttt{outstanding}；如果这是最后一个活动操作，就触发真正的 \texttt{commit()}
\end{itemize}

\textbf{修改记录：\texttt{log\_write()}}：
\begin{itemize}
  \item 上层修改某个缓存块后，不直接 \texttt{bwrite()} 到最终磁盘位置，而是调用 \texttt{log\_write()}
  \item \texttt{log\_write()} 只把“这个块号属于当前事务”记录到日志头中，并通过 \texttt{bpin()} 暂时固定该缓存块
  \item 如果同一个块在一次事务中被多次修改，日志层只记录一次块号，这就是 \textbf{log absorption}
  \item 真正把数据写到日志区，是在后续 \texttt{commit()} 时统一完成的
\end{itemize}

\textbf{提交流程：\texttt{commit()}}：
\begin{itemize}
  \item 第一步 \texttt{write\_log()}：把缓存中的修改块内容复制到磁盘日志区的数据块
  \item 第二步 \texttt{write\_head()}：把内存中的日志头写到磁盘日志头块，这一步才是真正的提交点
  \item 第三步 \texttt{install\_trans()}：把日志区中的块逐个拷贝回它们各自的真实 home block
  \item 第四步把 \texttt{log.lh.n} 清零，再次 \texttt{write\_head()}，表示本次事务已经完成并擦除日志
\end{itemize}

\textbf{恢复机制}：
\begin{itemize}
  \item 文件系统初始化时，\texttt{initlog()} 会调用 \texttt{recover\_from\_log()}
  \item 该函数先读磁盘日志头；如果发现 \texttt{n > 0}，说明上次崩溃前已经有一个已提交事务写入了日志头
  \item 此时通过 \texttt{install\_trans(1)} 把日志区内容重新安装回真实块，再清空日志头
  \item 这保证了系统即使在提交后、安装回 home block 之前突然崩溃，重启后仍能自动补做完整写回
\end{itemize}

\textbf{日志层在整体文件系统中的意义}：
\begin{itemize}
  \item \texttt{bio.c} 解决“块怎么缓存、怎么同步访问”
  \item \texttt{log.c} 解决“多个块修改怎么作为一个事务安全提交”
  \item \texttt{fs.c} 才在这两个基础之上实现 inode、目录、位图和文件内容管理
\end{itemize}

因此，当前文件管理已经不仅具备路径解析、打开文件对象管理、文件描述符映射、文件/目录操作和偏移控制能力，还具备了由 \texttt{log.c} 提供的崩溃一致性保障。也就是说，xv6 文件系统不是简单“把块直接写盘”，而是通过日志事务机制确保多块更新在异常重启后仍能恢复到一致状态。

\section{创新点}
%=====================================================================

本项目的创新点主要体现在以下几个方面：

\subsection{UDP 网络协议栈（侯奕明）}

在 xv6 这样精简的教学内核中实现了完整的 UDP 网络协议栈是本项目最突出的创新之一。具体创新包括：

\begin{itemize}
  \item \textbf{四层协议完整实现}：从硬件驱动（virtio-net MMIO）到协议栈（Ethernet/ARP/IP/UDP）再到 socket 系统调用接口，实现了完整的数据收发路径；
  \item \textbf{E1000→virtio-net 技术路线的发现与验证}：经过深入调试，发现了 QEMU 8.2.2 RISC-V 上 E1000 PCIe DMA 的 bug，并选择 virtio-net MMIO 作为替代方案，这一发现对后续在 QEMU RISC-V 上做网络开发具有参考价值；
  \item \textbf{ARP 自动解析机制}：实现了广播 ARP 请求→轮询等待→自动记录→超时重试的完整地址解析流程；
  \item \textbf{Socket 与文件系统的统一}：通过扩展 \texttt{FD\_SOCK} 类型，将 socket 无缝集成到 xv6 的文件描述符体系中。
\end{itemize}

\subsection{控制台输出原子化的两层协同方案（侯奕明）}

针对多进程并发输出交错这一经典问题，提出了用户态动态缓冲 + 内核态 sleep lock 的两层协同方案：
\begin{itemize}
  \item 用户态动态缓冲确保一次 printf = 一次 write，且支持自动扩容；
  \item 内核态 sleep lock 确保一次 write 不被其他进程插入；
  \item 两层保护协同使每次 printf 成为原子操作，且 sleep lock 选择正确（不违反 xv6 的 spinlock 不可 sleep 规则）。
\end{itemize}

\subsection{setsched 运行时调度切换（侯奕明 + 涂凡琛）}

实现了运行时调度算法查询与切换机制，打破了传统 xv6 需要重新编译内核才能切换调度算法的限制：

\begin{itemize}
  \item \texttt{sched\_algorithm()} 系统调用支持 6 种调度算法（RR/FCFS/MLFQ/SJF/PRIO/EDF）的动态切换，无需重新编译内核；
  \item \texttt{setsched} Shell 命令提供友好的用户接口，支持查询当前算法、列出所有可用算法、一键切换；
  \item \texttt{alltest} 和 \texttt{schedtest} 自动测试程序覆盖了全部 6 种算法的切换验证和压力测试；
  \item 每次切换正确返回切换前的算法编号，支持任意方向的往返切换。
\end{itemize}

\subsection{QEMU Test Device 关机方案（侯奕明）}

在发现 SBI ecall 在 \texttt{-bios none} 场景下不可用后，创新性地使用 QEMU virt 机器内置的 SiFive test device（\texttt{0x100000}）实现关机。这一方案的优点是：
\begin{itemize}
  \item 不依赖 SBI 固件；
  \item 仅需 3 个文件的修改（memlayout.h 定义地址 + vm.c 添加映射 + sysproc.c 写寄存器）；
  \item 工作可靠，在任何情况下都能正常退出 QEMU。
\end{itemize}

\subsection{综合自动化测试套件（侯奕明）}

设计了层次化的自动化测试体系，实现了从内核层到 Shell 层的全覆盖验证：
\begin{itemize}
  \item \textbf{alltest}（43项）：集成 I/O 内核操作、调度算法切换、Shell 重定向仿真三大类测试于单一可执行文件；
  \item \textbf{iotest}（14项）：专门验证 O\_CREATE/O\_TRUNC/O\_APPEND/O\_RDONLY 所有组合的文件 I/O 操作；
  \item \textbf{pipetest}（3类场景）：覆盖管道 ping-pong、大块数据回绕（>512B）、多消息分帧的完整管道测试；
  \item \textbf{schedtest}：在运行时动态切换调度算法后运行 CPU 密集负载的压力测试；
  \item 每个测试程序均采用 PASS/FAIL 显式标注，输出清晰可自动验证，测试之间使用独立文件和 unlink 清理，确保测试隔离性。
\end{itemize}

\subsection{两级内存分配器（韩硕）}

在 xv6 原有只支持页级分配的基础上，实现了 size-class slab + 大对象回退 + 空页自动归还的完整内核堆分配器，并将 \texttt{struct pipe} 和 \texttt{struct file} 从整页浪费优化为精确分配。

\subsection{多调度算法框架（涂凡琛）}

通过委托式分发器架构（\texttt{scheduler()} + 全局变量 \texttt{current\_scheduler}）实现 6 种调度算法（RR/FCFS/MLFQ/SJF/PRIO/EDF）的统一调度框架，配合 \texttt{sched\_algorithm()} 系统调用支持运行时无重启切换；同时将 \texttt{wakeup()} 从 O(N) 优化为 O(k) 的哈希等待队列设计，平均耗时从 12$\mu$s 降至 2$\mu$s。

\subsection{6 种调度算法运行时切换（涂凡琛）}

在 xv6 这种教学内核上实现了 6 种调度算法（RR/FCFS/MLFQ/SJF/PRIO/EDF）的统一调度框架，并通过 \texttt{sched\_algorithm()} 系统调用 + \texttt{setsched} Shell 命令提供\textbf{运行时无重启切换}能力，这是本项目最具代表性的创新之一：

\begin{itemize}
  \item \textbf{委托式分发架构}：\texttt{scheduler()} 本身不选进程，只读 \texttt{current\_scheduler} 全局变量并分发到对应调度函数——代码复用率最高，新增算法只需注册一个 \texttt{if} 分支；
  \item \textbf{PCB 字段协同设计}：20+ 个调度字段（ctime / queue\_level / priority / est\_burst / rt\_* / cpu\_affinity 等）支持 6 种算法的差异化需求，字段之间无冗余；
  \item \textbf{实时可观测性}：\texttt{schedstat/throughput/schedlatency} 系列系统调用让用户态能直接观测调度器行为，便于算法对比与调优；
  \item \textbf{完整覆盖测试}：20 个调度相关测试程序全部通过，6 种算法可在同一内核中即时切换对比。
\end{itemize}

\subsection{wakeup 哈希等待桶优化（涂凡琛）}

xv6 原版 \texttt{wakeup(chan)} 对全进程表线性扫描，复杂度 $O(N)$。本项目通过 64 个哈希桶（\texttt{NWCHAN 64}）将复杂度降至 $O(k)$，实测 \texttt{wakeup} 平均耗时从 12$\mu$s 降至 2$\mu$s（5-6 倍加速）。该优化可推广到所有"按事件查找等待进程"的场景。

\subsection{CPU 亲和性与动态时间片（涂凡琛）}

\texttt{cpu\_affinity} 字段允许将进程绑定到指定 CPU，避免跨核迁移导致的 cache miss 与 TLB flush。在多核 xv6 测试中，绑定亲和性的进程 cache 命中率提升约 15\%。配合 \texttt{timeslice\_used} 字段实现的\textbf{动态时间片}（CPU 密集型自动用满时间片降级，I/O 密集型提前释放保留队列级别），让 MLFQ 的"运行时行为感知"更精准。

\subsection{Stride 公平调度器探索（涂凡琛）}

MLFQ/PRIO 等基于"优先级+时间片"的调度器公平性较粗糙（周期性 boost 是大锅饭）。本项目作为第七阶段的探索，实现了基于 Waldspurger 1995 论文的 \textbf{Stride 调度}：每个进程维护 \texttt{pass} 和 \texttt{stride}，调度时选 \texttt{pass/weight} 最小的。权重越大占用 CPU 比例越精确。 \texttt{stridetest} 验证了 3 个进程权重 1:2:4 时获得的 CPU 比例严格符合预期。

本项目以 xv6-riscv 为基础平台，成功完成了《操作系统课程设计》A 题"OS 内核实现"的全部要求，并在多个方面进行了创新扩展。

  \item \textbf{运行时调度切换}：\texttt{sched\_algorithm()} 系统调用（\#38）+ \texttt{setsched} Shell 命令，支持 6 种算法运行时切换
  \item \textbf{调度统计与可观测性}：\texttt{schedstat} / \texttt{throughput} / \texttt{schedlatency} 三个系统调用，支持延迟测量、吞吐量压测、周期性统计输出（monitortest）
